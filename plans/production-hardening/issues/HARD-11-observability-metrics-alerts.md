# HARD-11: Observability: business metrics + alert pack

> Priority: P2 · Adversarial verdict: **REVISE** · Migration: No
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Add an IBusinessMetrics abstraction (Application.Abstractions) with an App Insights TelemetryClient-backed implementation (GetMetric pre-aggregation, exempt from adaptive sampling) and a no-op fallback, registered in AddReplyInMyVoiceInfrastructure via a TelemetryClient-conditional factory so Functions/Worker/Api/tests all resolve it. Emit 7 business metrics from existing code paths only (outbox dispatcher handler, Stripe webhook handler, rewrite job handler, provider HTTP resilience handler), all at the Application/DI/HttpClient layer — zero engine-internal coupling, so the planned engine swap keeps every metric. Service Bus DLQ depth uses the native Azure Monitor DeadletteredMessages metric (no timer probe — justified below). Ship an idempotent az CLI script under infra/alerts/ creating 5 scheduled-query alerts on customMetrics plus 1 native metric alert on the queue DLQ; the script is never run in CI.

8. DLQ DEPTH — DECISION: native Azure Monitor metric alert, NO timer probe. Justification (document verbatim in infra/alerts/README.md): (a) ServiceBusAdministrationClient.GetQueueRuntimePropertiesAsync requires Manage rights; the Function app's connection should stay least-privilege send/listen, and widening it for a metrics probe inverts the security tradeoff; (b) Azure Monitor natively emits DeadletteredMessages for Microsoft.ServiceBus/namespaces with an EntityName dimension at 1-minute granularity — zero code, zero polling cost, zero new failure modes, and it keeps working across engine swaps and even a Functions outage (which is precisely when the DLQ alert matters most); (c) a timer probe that itself fails silently would need its own watchdog. Per-issue tradeoff: native metric cannot run custom logic, but depth>0 alerting needs none.

Rules: [1] rimv-outbox-backlog-age, scheduled query on $APPINSIGHTS_ID, KQL: customMetrics | where name == "outbox_backlog_age_seconds" | summarize AggregatedValue = max(valueMax), condition AggregatedValue > 120, window 10m, eval 5m, severity 2 — dispatcher SLA is 15s, 2 minutes of un-dispatched head-of-queue = stuck dispatcher or poison message. [2] rimv-outbox-failed: customMetrics | where name == "outbox_failed_total" | summarize AggregatedValue = sum(valueSum), > 0 over 15m, eval 5m, sev 2 — handler failures are retried but any failure is worth eyes (rewrite jobs are latency-sensitive). [3] rimv-stripe-event-failed: same shape for stripe_event_failed_total, > 0 over 15m, eval 5m, sev 1 — billing correctness; Stripe retries mask user impact but repeated failures risk grant loss. [4] rimv-quota-release-spike: quota_released_total sum > 10 over 30m, eval 15m, sev 2 — releases are per-failure by design (never-charge rule); a spike means provider outage or quality regression; rewrite_quality_failure_total stays dashboard/diagnosis-grade (its error_code split tells which gate). [5] rimv-provider-circuit-open: provider_breaker_open_total sum >= 3 over 15m, eval 5m, sev 2 — single opens are tolerated noise (circuit state resets on HttpClientFactory handler rotation ~2min), sustained opens = provider down. [6] rimv-servicebus-dlq: az monitor metrics alert (create-or-update) scoped to $SERVICEBUS_NAMESPACE_ID, --condition "max DeadletteredMessages > 0 where EntityName includes $SERVICEBUS_QUEUE_NAME", window 5m, frequency 1m, sev 1 — any dead-lettered rewrite job is a lost paid-user request until replayed. README.md documents each threshold's rationale, the DLQ decision (step 8), monthly cost estimate (~5 scheduled-query rules ≈ US$7.5 + metric alert ≈ US$0.10), and an operator checklist: az login, correct subscription, and verify the Function app has APPLICATIONINSIGHTS_CONNECTION_STRING set by NAME via az functionapp config appsettings list (value must not be printed).

## Risks

- HARD-07 collision: this design relocates and parameterizes ProviderHttpResilienceHandler (currently private nested in ServiceCollectionExtensions.cs:454-653). If HARD-07 lands first and already moved/refactored the circuit, the Codex worker must implement ONLY the contract 'metrics.Record(provider_breaker_open_total, 1, client_name, <name>) on the closed-to-open transition' at wherever that transition lives, and skip the relocation steps. Supervisor must sequence or rebase these two issues.
- InfrastructureServiceCollectionTests (uses System.Reflection and Microsoft.Extensions.Http) pins DI/provider-selection internals; the handler relocation and the AddHttpMessageHandler(sp => ...) change may break a pin not fully read here — the worker must run the FULL suite and update pins in the same PR (a narrow --filter also skips AdminRouteMetadataTests).
- GetMetric identity rule: a metric name used with a dimension must never be called without one (and vice versa) or the SDK throws (swallowed by the adapter, but the series silently vanishes). Mitigated by BusinessMetricNames being the single source of truth with the dimension signature documented per name.
