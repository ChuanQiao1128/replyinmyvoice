# HARD-04: Handle missing Stripe lifecycle events

> Priority: P0 · Adversarial verdict: **PASS** · Migration: No
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Extend ProcessStripeWebhookHandler (the single Stripe inbox processor) to handle three additional lifecycle events: invoice.payment_action_required (SCA/3DS challenge -> upsert StripeInvoice, grace-style PastDue state for renewal invoices, user notification), customer.deleted (clear StripeCustomerId/StripeSubscriptionId, set Inactive, clear grace, write AdminAuditLog row), and customer.source.expiring (card-expiry early warning email for users with a recurring subscription). New notifications flow through the existing OutboxMessage table + DispatchDueOutboxHandler (two new IOutboxMessageHandler implementations), not the swallowed post-commit path, satisfying the repo guardrail "no direct post-commit publishing". payment_intent.payment_failed is deliberately NOT handled (rationale in notes); all other unhandled events keep the current explicit-ignore behavior (_ => null in SyncPayloadAsync). No EF migration needed.

## Implementation approach

1. DTO (StripeWebhookPayloadDto.cs): append 5 optional params to the StripeWebhookObjectDto positional record (after InvoicePdf, all defaulted null so every existing constructor call compiles): `string? BillingReason = null, string? CardBrand = null, string? CardLast4 = null, int? CardExpMonth = null, int? CardExpYear = null`. In ParseObject add: BillingReason from GetString(stripeObject, "billing_reason"); CardBrand from GetString(stripeObject, "brand"); CardLast4 from GetString(stripeObject, "last4"); CardExpMonth/CardExpYear via GetLong(stripeObject, "exp_month"/"exp_year") cast to int? (null when absent — do NOT use the existing GetInt32 which defaults to 0). Note: on customer.deleted the payload object IS the customer, so the customer id arrives in Object.Id, not Object.CustomerId; on customer.source.expiring the card object's "customer" property maps to the existing CustomerId field. No parser changes needed for those two.

2. ProcessStripeWebhookHandler.cs: add ctor params `IOutboxMessageRepository outboxMessages, IAdminUserRepository adminUsers` (DI registrations already exist at ServiceCollectionExtensions.cs:84/96). Add a private static `OutboxJsonOptions = new JsonSerializerOptions { PropertyNamingPolicy = JsonNamingPolicy.CamelCase }` mirroring CreateRewriteAttemptHandler.cs:24-27. Extend the SyncPayloadAsync switch (line 92-107) with three arms BEFORE the `_ => null` default, which stays unchanged so all other events remain explicitly ignored:
   "invoice.payment_action_required" => SyncInvoicePaymentActionRequiredAsync(payload, now, ct)
   "customer.deleted" => SyncCustomerDeletedAsync(payload, now, ct)
   "customer.source.expiring" => SyncCustomerSourceExpiringAsync(payload, now, ct)

3. SyncInvoicePaymentActionRequiredAsync — mirror SyncInvoicePaymentFailedAsync (lines 162-204) with these differences:
   a. Resolve customerId/subscriptionId; both null -> return null (ignore). User via FindByStripeCustomerOrSubscriptionAsync; null -> return failure string "No matching user for Stripe invoice payment_action_required customer {x} subscription {y}." (replayable, consistent with payment_failed).
   b. Call existing UpsertStripeInvoiceAsync(user, "invoice.payment_action_required", stripeObject, now, ct); propagate failure string. (ResolveStripeInvoiceStatus already falls back to the object's own "open" status — no change there.)
   c. Sync user.StripeCustomerId/StripeSubscriptionId as payment_failed does.
   d. Grace-style state ONLY when subscriptionId is non-null AND Normalize(stripeObject.BillingReason) != "subscription_create" (an incomplete first payment is completed inside Checkout's own 3DS UI and must not flip a not-yet-active user to PastDue): set user.SubscriptionStatus = SubscriptionStatus.PastDue; user.PaymentFailedAt ??= now; user.PaymentGraceEndsAt ??= ResolvePaymentGraceEndsAt(stripeObject, user.CurrentPeriodEnd, now). Use ??= (NOT plain assign) and do NOT null PaymentGraceReminderSentAt — action_required typically precedes payment_failed for the same invoice and must not restart an already-running grace/reminder window. Always set UpdatedAt = now and RowVersion = Guid.NewGuid().
   e. ALWAYS (whether or not grace applied) add an OutboxMessage inside the same transaction via outboxMessages.AddAsync: MessageType "StripePaymentActionRequiredNotification", PayloadJson = camelCase serialize of private record StripePaymentActionRequiredPayload(Guid UserId, string? InvoiceId, string? HostedInvoiceUrl), Status Pending, CreatedAt/NextAttemptAt = now, MaxAttempts = 10, CorrelationId = payload.EventId. Recovery already works: the existing invoice.payment_succeeded handler clears grace and flips PastDue -> Active.

4. SyncCustomerDeletedAsync:
   a. customerId = Normalize(payload.Object.Id); null -> return null.
   b. user = appUsers.GetByStripeCustomerIdAsync(customerId, ct); null -> return null (explicitly ignored: end-state "no user holds this customer id" already true; this also makes distinct-event replays naturally idempotent — second event finds no user, writes nothing).
   c. Capture previous values, then: user.StripeCustomerId = null; user.StripeSubscriptionId = null; user.SubscriptionStatus = SubscriptionStatus.Inactive; ClearPaymentGrace(user); user.UpdatedAt = now; user.RowVersion = Guid.NewGuid().
   d. await adminUsers.AddAuditLogAsync(new AdminAuditLog { AdminExternalAuthUserId = "system:stripe-webhook", AdminEmail = "system@replyinmyvoice.com", Action = "stripe.customer.deleted", TargetUserId = user.Id, DetailsJson = JsonSerializer.Serialize(new { stripeEventId = payload.EventId, stripeCustomerId = <previous>, stripeSubscriptionId = <previous>, previousSubscriptionStatus = <previous>.ToString() }, OutboxJsonOptions), CreatedAt = now }, ct). Action length 120 / AdminExternalAuthUserId 160 limits (AppDbContext.cs:441-443) are respected. NO user email by design (the billing identity is gone; this event commonly follows account erasure — emailing would confuse; the audit row is the durable record). Return null.

5. SyncCustomerSourceExpiringAsync:
   a. customerId = Normalize(payload.Object.CustomerId); null -> return null. user = GetByStripeCustomerIdAsync; null -> return null (no replay value for an expiry warning aimed at an unknown user — decision documented).
   b. Notify only when user.SubscriptionStatus is Active, Trialing, or PastDue (one-time pack buyers have no recurring charge; a card-expiry warning is noise). Otherwise return null.
   c. Add OutboxMessage: MessageType "StripeCardExpiringNotification", PayloadJson = camelCase serialize of private record StripeCardExpiringPayload(Guid UserId, string? Brand, string? Last4, int? ExpMonth, int? ExpYear), CorrelationId = payload.EventId, MaxAttempts 10. No user-row mutation. Idempotency = StripeEvent inbox dedup per EventId (BeginProcessingAsync returns null on replay, so the outbox row is created at most once per event).

## Risks

- Stripe event-type coverage (verified against API knowledge, runtime-unverifiable from this repo): invoice.payment_action_required and customer.deleted are standard webhook events. customer.source.expiring is real but documented for legacy Card/Source objects attached to customer.sources; for cards saved as PaymentMethods via Checkout (this product's only saved-card path), Stripe is NOT documented to emit it, and no payment_method.*.expiring event exists — so the card-expiry warning may rarely or never fire for real customers. The handler is still correct and harmless; a reliable early warning for PaymentMethod cards would need a scheduled job querying default payment methods via the Stripe API (out of scope, listed in notes).
- The Stripe webhook endpoint's enabled_events list (dashboard/API config, not in source) must include the three new event types or the code is dead on arrival. This is an ops step OUTSIDE the PR — do NOT touch STRIPE_WEBHOOK_SECRET or live routing; surface it in the PR description for the owner.
- Webhook ordering is not guaranteed: customer.subscription.updated(past_due) and invoice.payment_failed may land before/after invoice.payment_action_required. The ??= grace assignments and the existing handlers' own ??=/Clear logic make any interleaving converge, but the user may receive both an action-required and a failed-payment email for one invoice (~minutes apart). Accepted; dedupe would need new schema.
- customer.deleted commonly arrives after our own DeleteAccountHandler erases the account; if the user row is already gone or unlinked, the new handler ignores the event (by design) — no audit row is written for that path. The deletion is already audited by the account-deletion flow.
- Email delivery failures inside the outbox handler only trigger outbox retry if the notification path THROWS; NotificationService returns a result object for provider-level soft failures (Resend non-2xx may not throw — depends on ResendNotificationEmailProvider internals, not line-audited). At-least-once dispatch is guaranteed; at-least-once delivery is not. Same exposure as every existing notification; do not fix here.
- ProcessStripeWebhookHandler gains 2 ctor params: only the two test-file private factories construct it directly (verified by grep — DI everywhere else), but any unmerged branch constructing it will conflict.
- StripeWebhookObjectDto is a positional record used widely with named args; appending defaulted params is source-compatible but a Codex worker inserting them mid-list would break call sites — must append at the END.
- Engine-swap safety (constraint 5): zero contact — this design touches only Stripe webhook ingestion, notifications, and the outbox; no IRewriteEngineClient/provider/ResultJson surface.

## Design notes

Decisions inside the mandate: (1) payment_intent.payment_failed is intentionally NOT handled — subscription failures are already covered by invoice.payment_failed (grace + notification), and a failed payment inside an active Checkout session never completes the session, so the user sees the failure in Checkout UI; handling it would double-notify with no new DB effect. It stays explicitly ignored via the existing default switch arm, and a test pins that. (2) customer.deleted sends no user email — the audit row is the record; mandate's per-item spec ('clear/mark + audit log') wins over the generic 'each: + notification' clause. (3) New notifications use the outbox (guardrail: no direct post-commit publishing) rather than the existing swallowed post-commit pattern; the four EXISTING IStripeEventNotifier flows are left as-is to keep scope tight. Adjacent problems observed but OUT of scope: (a) post-commit notification exceptions at ProcessStripeWebhookHandler.cs:533-547 are swallowed silently for the four existing flows — migrating them to the same outbox pattern is a natural follow-up; (b) webhook responses always return HTTP 200 even when the event is recorded Failed (StripeWebhookFunction.cs:139), so Stripe never auto-retries 'No matching user' races — replay is manual; (c) StripeReconciliation covers only PaymentIntents vs credit grants, not subscription state drift — customer.deleted/cancel drift would not be caught by it; (d) the customer.source.expiring gap for PaymentMethod-saved cards could be closed later with a daily timer that lists default payment methods expiring within 30 days via the Stripe API and enqueues the same 'StripeCardExpiringNotification' outbox message (reuses everything this design adds). Owner/ops prerequisite recorded in risks: add the three event types to the webhook endpoint's enabled_events. All new strings/identifiers avoid the five CI-banned substrings.
