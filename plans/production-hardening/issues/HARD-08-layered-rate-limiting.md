# HARD-08: Layered rate limiting

> Priority: P1 · Adversarial verdict: **PASS** · Migration: **YES** — additive EF migration required
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Add layered rate limiting around the rewrite surface without touching the engine boundary: (1) a per-user per-minute DB-window limit (default 6/min, env-configurable) on consumer POST /api/rewrite in both the Functions host and the Api test host, returning 429 + Retry-After + X-RateLimit-* and checked BEFORE quota reservation so limited requests never consume quota; (2) an in-process singleton pre-check gate decorating IApiKeyRateLimiter so v1 floods are shed cheaply per-instance while the ApiKeyRateLimitWindow DB row stays authoritative (zero call-site changes — pure DI decorator); (3) a Cloudflare WAF rate-rule config artifact under infra/cloudflare/ (JSON payload + doc, never applied by CI). Frontend proxy maps backend 429 to a friendly {code:"rate_limited"} payload and forwards the standard headers via the existing copyV1ResponseHeaders helper.

## Implementation approach

LAYER 1 — consumer per-user DB window (multi-instance safe by construction; same pattern as v1).

1. Create Domain/Entities/UserRewriteRateLimitWindow.cs as an exact structural mirror of ApiKeyRateLimitWindow.cs but keyed by UserId (FK to AppUser, nullable nav `AppUser? User`).
2. Register it in AppDbContext (DbSet + config block cloned from lines 401-411, substituting UserId; `.WithMany()` with no collection property on AppUser so AppUser.cs is untouched). EF's conventional index name will be IX_UserRewriteRateLimitWindows_UserId_WindowStart — hardcode that exact string in the limiter's race-exception matcher.
3. Generate migration: `dotnet ef migrations add AddUserRewriteRateLimitWindows --project backend-dotnet/src/ReplyInMyVoice.Infrastructure --startup-project backend-dotnet/src/ReplyInMyVoice.Infrastructure --context AppDbContext`. Verify the generated file contains only CreateTable/CreateIndex (no Drop*/Alter*). SQLite tests use EnsureCreated so no SQLite branch needed; the limiter does no date arithmetic in SQL (window computed in C#, equality match only), matching ApiKeyRateLimiter.
4. Create Infrastructure/Services/UserRewriteRateLimiter.cs. Interface `IUserRewriteRateLimiter { bool Enabled {get;} int LimitPerMinute {get;} Task<ApiKeyRateLimitResult> CheckAndIncrementAsync(Guid userId, DateTimeOffset now, CancellationToken ct); }` (reuses the existing public ApiKeyRateLimitResult record from ApiKeyRateLimiter.cs — same namespace, no new DTO). Implementation is a line-for-line port of ApiKeyRateLimiter.CheckAndIncrementCoreAsync against db.UserRewriteRateLimitWindows: Serializable transaction via CreateExecutionStrategy, insert-or-increment, MaxAttempts=5 with 10ms*attempt backoff, catch DbUpdateConcurrencyException / unique-index race / SQLite busy, unexpected exception => ApiKeyRateLimitResult.Unavailable. Constructor `UserRewriteRateLimiter(Func<AppDbContext> dbContextFactory, int limitPerMinute)`; `Enabled => limitPerMinute > 0` (NOTE: unlike ApiKeyRateLimiter, non-positive means DISABLED for the consumer surface, not fail-closed — call sites skip the check when !Enabled).
5. DI in ServiceCollectionExtensions.cs (next to line 164): `var consumerRewriteLimit = int.TryParse(configuration["CONSUMER_REWRITE_RATE_LIMIT_PER_MINUTE"], out var parsedConsumerLimit) ? parsedConsumerLimit : 6;` then `services.AddScoped<IUserRewriteRateLimiter>(sp => new UserRewriteRateLimiter(sp.GetRequiredService<Func<AppDbContext>>(), consumerRewriteLimit));`. Never log/print values; both new env vars are optional-with-default so ValidateReplyInMyVoiceRuntimeConfiguration is NOT modified.
6. Wire RewriteHttpFunctions.CreateRewriteAttempt (Functions host, the prod path): add `IUserRewriteRateLimiter userRateLimiter` to the primary constructor. After the getOrCreateUserHandler call (line 89-91) and BEFORE plan/createRewriteAttemptHandler insert:
   - if (!userRateLimiter.Enabled) skip entirely;
   - result = await userRateLimiter.CheckAndIncrementAsync(user.Id, DateTimeOffset.UtcNow, ct);
   - IsUnavailable => return FunctionHttpResults.Problem("Rate limit check unavailable", "Rewrite limit could not be checked. Please retry shortly.", StatusCodes.Status503ServiceUnavailable, "rate_limit_unavailable") — fail closed, NO quota touched;
   - IsLimited => SetRateLimitHeaders(request.HttpContext.Response, result, includeRetryAfter: true, now) then return FunctionHttpResults.Problem("Too many requests", "Rewrite rate limit reached. Please wait a moment and retry.", StatusCodes.Status429TooManyRequests, "rate_limited") (coded {error:{code,message}} envelope per FunctionHttpResults.cs);
   - Allowed => SetRateLimitHeaders(..., includeRetryAfter: false, now) and continue to quota reservation.
   SetRateLimitHeaders is a private static copy of V1RewriteHttpFunctions.cs:705-719 (X-RateLimit-Limit / X-RateLimit-Remaining / X-RateLimit-Reset unix-seconds / Retry-After via result.RetryAfterSeconds). Do NOT modify V1RewriteHttpFunctions.cs.
7. Mirror exactly the same block in Api/Program.cs MapPost("/api/rewrite") (insert after line 181, before line 182's plan lookup), injecting IUserRewriteRateLimiter as a handler parameter and reusing the file's existing SetV1RateLimitHeaders helper (line 1174) and {error:{code,message}} envelope. This keeps the WebApplicationFactory test host behaviorally identical to the Functions host.

LAYER 2 — v1 in-process approximate pre-check (DB stays authoritative).

8. Create Infrastructure/Services/InProcessRateLimitGate.cs: singleton, `private readonly object gate` + `Dictionary<Guid,(DateTimeOffset WindowStart, int Count)>`. ShouldShed(key, limit, now, out resetAt): compute the same UTC-minute window start as ApiKeyRateLimiter.ToMinuteWindowStart; reset the per-key counter when the stored window differs; increment; return Count > limit. resetAt = windowStart.AddMinutes(1). When the dictionary exceeds 10000 entries, drop entries whose WindowStart != current window. Correctness note to put in the class doc-comment: the per-instance arrival count is a lower bound on total arrivals for the key in that minute, and the v1 endpoints call the limiter immediately after auth before any other rejection, so by the time this instance sees arrival limit+1 the authoritative DB window has already reached the limit — shedding returns the same 429 the DB would. It is approximate only in that (a) other instances' traffic is not counted (those requests still hit the DB) and (b) if the DB was unavailable for earlier arrivals a shed converts a would-be 503 into a 429 (both refuse without consuming quota).
9. Create Infrastructure/Services/PreCheckedApiKeyRateLimiter.cs implementing IApiKeyRateLimiter: `if (rateLimitPerMinute <= 0) return inner.CheckAndIncrementAsync(...)` (preserves the existing fail-closed Limited(0,...) semantics); `if (gate.ShouldShed(apiKeyId, rateLimitPerMinute, now, out var resetAt)) return ApiKeyRateLimitResult.Limited(rateLimitPerMinute, rateLimitPerMinute, resetAt);` else delegate. Remaining computes to 0 so existing X-RateLimit-Remaining=0 assertions hold.

## Risks

- Minute-boundary flake in window tests — mitigated by seeding both the current and next minute window in the authoritative-DB test and by passing explicit DateTimeOffset values in unit tests; do not sleep across boundaries
- Consumer endpoint logic is duplicated across two hosts (Functions/RewriteHttpFunctions.cs = prod, Api/Program.cs = WebApplicationFactory test host); a worker could wire only one — both call sites are named explicitly and ConsumerRewriteRateLimitTests pins the Api host while RewriteHistoryTests compiles the Functions ctor change; there is still no HTTP-level test of the Functions host (pre-existing gap for all consumer routes)
- Pre-check shed can return 429 where the pure-DB path would have returned 503 rate_limit_unavailable (when the DB went down mid-minute after some arrivals); both refuse without consuming quota — documented in the gate doc-comment, accepted approximation
- Idempotent replays of POST /api/rewrite count against the per-user limit (check sits before the handler that detects replays); default 6/min is generous vs the UI's single-submit flow, and polling uses GET which is not limited — accepted
- ApiKeyRateLimitWindows already accumulates one row per key-minute with no purge; UserRewriteRateLimitWindows inherits the same unbounded-growth pattern (flagged in notes, out of mandate scope)
- Reusing the ApiKeyRateLimitResult record for the consumer limiter couples the two surfaces' result shape; it is additive-only and already public in Infrastructure.Services — renaming it later would be a separate refactor
- Unknown (not verifiable from source): live Function app instance count and whether the Worker fronts ALL backend traffic (the WAF artifact assumes /api/* routes through the replyinmyvoice-app Worker zone); the WAF rule is an unapplied artifact so a wrong expression has zero blast radius until the owner applies it manually
- Unknown: whether the Cloudflare zone's plan tier supports the response-content override on the rate-limit block action (paid feature on some tiers); the .md must note the fallback (default 429 HTML) — artifact-only, no runtime impact
- The 429 passthrough previously fell into the generic !response.ok branch of app/api/rewrite/route.ts; any consumer that depended on receiving the raw C# ProblemDetails for 429 would see the new shape — no such consumer exists today (workspace showed a generic message), and the new shape gives a readable error string

## Design notes

Engine-swap safety (hard constraint 5): every piece of this design lives at the HTTP-function, DI-registration, or schema layer. IRewriteEngineClient, IRewriteProvider, FactReconstructRewriteProvider, ProcessRewriteJobHandler, ProviderHttpResilienceHandler and all ResultJson consumers are untouched; an engine swap carries all three layers unchanged. Design choice justification per mandate: the DB-window option was chosen for layer 1 (over a leaner in-memory-only approach) because the Functions app can scale out and the repo already has a proven, test-hardened implementation of exactly this pattern (ApiKeyRateLimiter + V1RewriteRateLimitTests) — consumer POST volume is quota-bound and tiny, so one extra Serializable transaction per submit is negligible; the leaner in-process technique is instead applied where it pays off (layer 2, in front of the hot v1 path) as a pure decorator so the DB row remains the single source of truth. Adjacent problems noticed but deliberately OUT of scope: (1) no retention/purge for ApiKeyRateLimitWindows (and now UserRewriteRateLimitWindows) — a follow-up could extend RetentionService to delete windows older than 24h; (2) GET /api/rewrite-attempts/{id} and GET /api/me/* consumer endpoints remain unlimited (mandate covers POST /api/rewrite only); (3) rewrite-workspace.tsx titleForQualityFailure checks stale reason strings (lines 128-134) that never match real codes — pre-existing, unrelated; (4) ApiKey.CurrentPeriodUsage dual-accounting with ApiKeyUsage rows is untouched; (5) local-env.md could document CONSUMER_REWRITE_RATE_LIMIT_PER_MINUTE and API_RATE_LIMIT_PRECHECK_ENABLED — optional doc touch the worker may include if trivial. Codex brief reminders: never print env values; the two new env vars are optional-with-default so the fail-fast validator must NOT gain entries for them; all new tests must be serial-safe (assembly parallelization is off) and must use DI replacement instead of process-env mutation; the full suite (not a --filter run) is the acceptance gate because InfrastructureServiceCollectionTests and AdminRouteMetadataTests only guard drift in full passes.
