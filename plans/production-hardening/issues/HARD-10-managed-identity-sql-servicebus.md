# HARD-10: Managed Identity support for SQL + Service Bus

> Priority: P1 · Adversarial verdict: **REVISE** · Migration: No
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Add reversible, flag-gated Managed Identity support for Azure SQL and Service Bus at the DI/configuration layer. When USE_MANAGED_IDENTITY=true, the Service Bus publisher is built from a fully-qualified namespace + DefaultAzureCredential, and the SQL connection string is rewritten (or built) with Authentication=Active Directory Default (the EF-Core-8 / Microsoft.Data.SqlClient 5.x proven approach — no token interceptor, SqlClient manages token lifetime internally). Connection strings remain the default and the fallback; flag off is byte-identical to today. The ServiceBusTrigger needs zero code change (identity binding is purely the ServiceBus__fullyQualifiedNamespace app setting); a docs/managed-identity-rollout.md runbook ships the exact az CLI role assignments, T-SQL, app settings, and rollback. No infra is executed in this wave.

## Risks

- The live Function App's actual trigger connection setting name (ServiceBus vs ConnectionStrings__ServiceBus vs others) is not verifiable from source; the runbook's discovery step (appsettings list --query "[].name") resolves this before the trigger flip. If both a literal ServiceBus connection-string setting and ServiceBus__fullyQualifiedNamespace exist, the WebJobs extension keeps using the connection string — intended during staged rollout, but operators must complete step 8 for the consumer to actually switch.
- RBAC propagation (up to ~5 min) and SQL CREATE USER FROM EXTERNAL PROVIDER ordering: flipping USE_MANAGED_IDENTITY before roles/users exist causes runtime auth failures; runbook orders identity→roles→SQL user→flag and keeps connection strings as instant rollback. Health authMode makes the active mode observable.
- DefaultAzureCredential adds first-token latency (~1-2s) on cold start for both SQL and Service Bus; within existing timeout budgets (SQL ConnectTimeout 30s, 15min job TTL) but worth watching in App Insights after flip.
- Microsoft.Data.SqlClient is pinned transitively by EF Core 8.0.19 (5.1.x); SqlAuthenticationMethod.ActiveDirectoryDefault is supported there, but adding explicit Azure.Identity 1.13.x alongside SqlClient's older transitive Azure.Identity could surface NU1608/binding warnings — verify clean restore/build in CI (it normally just upgrades the transitive and silences known advisories).
- ServiceBusReadinessCheck gains a positional record parameter; any code constructing it must be updated in the same change (sole constructor call is HealthFunction.cs:150). JSON change is additive so the dotnet-azure.yml /api/health gate and any external monitors are unaffected.
- AppDbContextDesignTimeFactory.cs and the CI sqlserver-migration gate still use plain connection strings — intentionally unchanged; if someone later runs migrations under MI the SQL user would need ddladmin (explicitly out of scope, noted in the runbook).
- Worker project changes are build-verified only (Worker is not deployed); its ServiceBusClient MI path has no test coverage beyond compilation — acceptable since the Functions app is the production runtime.
- If prod sets USE_MANAGED_IDENTITY=true while DATABASE_URL still points at a connection string containing SQL credentials, the resolver silently strips them and switches auth — correct by design, but means the flag alone changes DB auth; the runbook calls this out so the SQL user is created first.

## Design notes

Adjacent problems noticed but kept out of scope: (1) AzureWebJobsStorage (Functions host storage) remains key-based — identity-based host storage (AzureWebJobsStorage__accountName + Storage Blob Data Owner) is a natural follow-up. (2) HealthFunction duplicates the 4-way Service Bus connection-string fallback chain that also appears in ServiceCollectionExtensions and both Worker classes; this design routes new logic through ManagedIdentityConfiguration but does not consolidate the legacy chain. (3) ValidateReplyInMyVoiceRuntimeConfiguration's SQL/Service Bus messages are growing unwieldy; a structured 'one of: ...' format would help but would churn test pins. (4) The Stripe/DeepSeek/Sapling key requirements in validation are untouched per constraint — a future engine swap must revisit that block separately (engine-boundary note: this design touches only DB/Service Bus DI registration, never IRewriteEngineClient, IRewriteProvider, providers, or ResultJson). (5) Key Vault references for the remaining secrets (STRIPE_SECRET_KEY etc.) via the same managed identity would complete the no-secrets-in-app-settings story; deliberately not bundled. Design choice log: rejected the EF access-token DbConnectionInterceptor approach because Microsoft.Data.SqlClient's built-in Active Directory Default handles token caching/renewal per pooled connection with zero custom lifetime code and is the documented EF-Core-8-on-Functions pattern; rejected making MI lose to a present connection string because flag-wins gives a one-setting rollback (USE_MANAGED_IDENTITY=false) without deleting namespace settings.
