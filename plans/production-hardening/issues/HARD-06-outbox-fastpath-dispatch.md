# HARD-06: Outbox fast-path dispatch after commit

> Priority: P1 · Adversarial verdict: **REVISE** · Migration: No
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Add a best-effort outbox fast path: after CreateRewriteAttemptHandler's reservation transaction commits, the handler asks a new IOutboxFastPathDispatcher to immediately claim-and-dispatch the just-written RewriteJobCreated OutboxMessage using the exact same claim/lock/MarkSent discipline as the 15s timer (a new claim-by-id primitive on OutboxMessageRepository plus a TryDispatchOneAsync method on DispatchDueOutboxHandler). Any failure is swallowed and logged; the row stays Pending/backoff-rescheduled and the existing OutboxDispatcherTimerFunction delivers it, so the outbox row remains the source of truth and the timer remains the guarantee. Because the hook lives in the application handler (post-commit, outside the transaction), all four entry points (Functions consumer POST /api/rewrite, Functions v1 POST /api/v1/rewrite, and both Api test-host endpoints) get the fast path with zero HTTP-layer changes, and p50 enqueue-to-publish drops from ~7.5s (timer midpoint) to one Service Bus send (~20–150ms), capped by a 5s timeout and an OUTBOX_FAST_PATH_ENABLED kill switch.

## Implementation approach

1. REPOSITORY PRIMITIVE. In IOutboxMessageRepository.cs add ClaimByIdAsync(Guid messageId, DateTimeOffset now, string lockedBy, TimeSpan claimLease, ct) returning Task<OutboxMessage?>. Implement in OutboxMessageRepository.cs: `var message = await db.OutboxMessages.AsTracking().SingleOrDefaultAsync(x => x.Id == messageId, ct);` return null if missing; then evaluate claimability IN MEMORY (no IsSqlite branch needed for a single PK row): claimable iff (Status == Pending || Status == Processing) && NextAttemptAt <= now && (LockedUntil is null || LockedUntil <= now); if not claimable return null; otherwise apply EXACTLY the mutation ClaimDueAsync applies (Status=Processing, LockedBy=lockedBy, LockedUntil=now+claimLease, LastAttemptAt=now, RowVersion=Guid.NewGuid()) and return the entity. RowVersion is the concurrency token, so a concurrent timer claim loses via DbUpdateConcurrencyException and retries.

2. DISPATCH REUSE. In DispatchDueOutboxHandler.cs extract the existing per-message loop body (handler-map lookup, handler.HandleAsync, MarkSentAsync+SaveChanges, catch-not-OCE -> MarkFailedAttemptAsync+SaveChanges) into `private async Task<bool> DispatchClaimedMessageAsync(OutboxMessage message, DateTimeOffset now, CancellationToken ct)` returning true only on MarkSent. HandleAsync's foreach becomes `await DispatchClaimedMessageAsync(message, command.Now, ct);` (return value ignored) — timer semantics byte-identical. Add `public async Task<bool> TryDispatchOneAsync(DispatchOutboxMessageCommand command, CancellationToken ct = default)`: claim inside unitOfWork.ExecuteInTransactionAsync(IsolationLevel.Serializable, ClaimRaceMaxAttempts) calling ClaimByIdAsync + SaveChangesAsync (only when claimed); if null return false; else return await DispatchClaimedMessageAsync(claimed, command.Now, ct) OUTSIDE the transaction — same shape as the timer path (claim in tx, publish/mark out of tx). Create DispatchOutboxMessageCommand.cs record (OutboxMessageId, Now, LockedBy). Note: ReplyInMyVoice.Worker/OutboxDispatcherWorker.cs only calls HandleAsync, so it compiles unchanged.

3. FAST-PATH WRAPPER. Create Application abstraction IOutboxFastPathDispatcher { Task TryDispatchAsync(Guid outboxMessageId, CancellationToken ct = default); } (Application has no logging reference, so the implementation lives in Infrastructure). Create Infrastructure/Services/OutboxFastPathDispatcher.cs: `public sealed record OutboxFastPathOptions(bool Enabled, TimeSpan Timeout);` and `public sealed class OutboxFastPathDispatcher(DispatchDueOutboxHandler dispatchDueOutboxHandler, OutboxFastPathOptions options, ILogger<OutboxFastPathDispatcher> logger) : IOutboxFastPathDispatcher`. TryDispatchAsync: if (!options.Enabled) return; Stopwatch.StartNew(); try { using var timeoutCts = new CancellationTokenSource(options.Timeout); /* deliberately NOT linked to caller ct: post-commit work must not abort on client disconnect */ var sent = await dispatchDueOutboxHandler.TryDispatchOneAsync(new DispatchOutboxMessageCommand(outboxMessageId, DateTimeOffset.UtcNow, $"{Environment.MachineName}:fastpath"), timeoutCts.Token); log Information "Outbox fast-path dispatched message {OutboxMessageId} in {ElapsedMs} ms." when sent, else Information "skipped ... timer dispatcher will deliver."; } catch (Exception ex) { logger.LogWarning(ex, "Outbox fast-path dispatch failed for message {OutboxMessageId}; timer dispatcher will retry.", outboxMessageId); } — catch-all, including OperationCanceledException: the method never throws.

4. HOOK AT THE WRITER. In CreateRewriteAttemptHandler.cs add ctor param `IOutboxFastPathDispatcher outboxFastPath`. Change HandleCoreAsync to return a private record CreateAttemptOutcome(ApplicationResult<RewriteAttemptDto> Result, Guid? OutboxMessageId): in both Created branches capture `var outboxMessage = CreateRewriteJobOutboxMessage(attempt.Id, command.Now);` before AddAsync and return its Id; every other return point (NotFound, QuotaExceeded, Conflict, Existing) returns null OutboxMessageId. ## Risks

- Added request latency: the fast path runs synchronously inside the HTTP request after commit (claim tx + Service Bus send, typically ~20–150ms; live Azure SQL claim adds a few ms). Bounded by OUTBOX_FAST_PATH_TIMEOUT_SEC (default 5s) and removable via OUTBOX_FAST_PATH_ENABLED=false without redeploy of behavior-critical code. CI cannot prove the prod <500ms p50; the log line + SentAt-CreatedAt query are the post-deploy measurement instrument.
- Duplicate publish window (pre-existing, slightly more likely): if the fast path publishes but crashes before MarkSent, the 30s lease expires and the timer republishes. The mandate names Service Bus consumer idempotence as the backstop — ProcessRewriteJobHandler.MarkProcessingAsync (line 214-237) only advances Pending attempts, and a Succeeded attempt is not re-finalized (line 252); this matches the identical window already present in the timer path (publish-then-mark).
- Serializable claim contention: fast path and a 15s timer tick can claim concurrently; resolved by RowVersion concurrency token + ClaimRaceMaxAttempts=5 retries + IsRetryableTransactionRace (deadlock/serialization matching in UnitOfWork.cs:85-103). Worst case the fast path loses and logs a warning; delivery falls back to the timer (correctness unchanged). The new OutboxFastPathDispatchTests must stay serial-safe (assembly parallelization already disabled).
- Shared scoped DbContext: the fast path reuses the request scope's AppDbContext after commit. Verified safe (entries Unchanged after SaveChanges; claim re-attaches the tracked OutboxMessage; failure path calls ChangeTracker.Clear() only inside the fast-path's own transaction, after the create has committed and the DTO is materialized). Codex must NOT resolve a fresh scope/context — that would break the SQLite shared-connection test fixtures.
- Behavior change to a pinned integration test: RewriteApiTests.Rewrite_creates_attempt_and_outbox_message flips from Pending to Sent — this is the intended new contract and must be updated in the same PR; no other Status pins exist after HTTP POSTs (verified by grep; QuotaUseCaseTests:53 pins ReserveQuotaHandler, which is untouched).
- Unverified from source: Worker/ServiceBusRewriteWorker.cs and the live Function app's APPLICATIONINSIGHTS_CONNECTION_STRING were not line-audited here; the Worker project only calls DispatchDueOutboxHandler.HandleAsync (signature preserved) so it compiles, and the measurement log line degrades to console logging if App Insights is absent.
- Cancellation semantics decision: the dispatcher deliberately ignores the caller's CancellationToken (standalone timeout CTS) so a client disconnect cannot strand a published-but-unmarked message; reviewers may flag this — it is intentional and documented in the class comment.

## Design notes

Design decisions and adjacent observations (out of scope, do not expand): (1) Hook placement — the fast path is invoked from CreateRewriteAttemptHandler.HandleAsync post-commit rather than from each HTTP function; this covers consumer, v1, and both Api test-host endpoints with one change and keeps the sanctioned evolution at the outbox layer (claim discipline, not a second publish path). The house rule 'no direct post-commit publishing' is honored: the row is written inside the transaction, the fast path goes through ClaimByIdAsync with the same lease/MarkSent discipline, and the timer remains the delivery guarantee. (2) Generalization came free: TryDispatchOneAsync works for any future OutboxMessage type (RewriteJobCreated is currently the only registered IOutboxMessageHandler, verified ServiceCollectionExtensions.cs:167); future writers just call IOutboxFastPathDispatcher with their row id. (3) Engine-swap safety: nothing under Infrastructure/Providers or Domain/RewriteEngine is referenced; the change sits entirely on the outbox/dispatch boundary, upstream of IRewriteEngineClient. (4) Adjacent problems noticed but NOT addressed here: the v1 sandbox path and webhook deliveries (WebhookDelivery table via DispatchDueWebhooksHandler) have their own latency profile and could get an analogous fast path later; StripeWebhookFunction's swallowed post-commit notification exceptions (~lines 533-547) remain a separate hardening item; DispatchDueOutboxHandler builds its handler dictionary per scoped instance (minor allocation, harmless). (5) LockedBy is set to '<machine>:fastpath' during the claim for DB-side observability, but MarkSentAsync clears it on success — measurement therefore relies on the Information log line and SentAt-CreatedAt deltas, not LockedBy. (6) Worker brief reminder: per repo convention all bool/int config reads stay inline in AddReplyInMyVoiceInfrastructure with parse-with-default and env-var NAMES only; never add the new keys to ValidateReplyInMyVoiceRuntimeConfiguration (they are optional), and never log configuration values.
