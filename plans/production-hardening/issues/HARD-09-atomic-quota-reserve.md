# HARD-09: Atomic conditional UPDATE for quota reserve

> Priority: P1 · Adversarial verdict: **PASS** · Migration: No
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Replace the Serializable SELECT-then-mutate quota hot path with single-statement conditional UPDATEs modeled on PromoCodeRepository.TryIncrementRedemptionCountAsync (PromoCodeRepository.cs:25-43): period reserve becomes `UPDATE UsagePeriods SET ReservedCount=ReservedCount+1 ... WHERE Id=@id AND UsedCount+ReservedCount < @quotaLimit`, credit consume becomes `UPDATE RewriteCredits SET AmountConsumed=AmountConsumed+1 WHERE Id=@id AND AmountConsumed < AmountGranted`, and affected-rows=0 falls through to credit then 402. Finalize/release/expire paths claim the UsageReservation via a conditional Pending-only status UPDATE before adjusting counters, removing the remaining read-modify-write. The surrounding transaction (attempt + reservation + outbox rows) stays, dropped from Serializable to ReadCommitted: the admission decision now happens at write time under row locks, the create races are covered by unique indexes already classified retryable in UnitOfWork.IsRetryableTransactionRace (UnitOfWork.cs:85-104), and Microsoft.Data.Sqlite promotes ReadCommitted to Serializable (verified empirically against the repo's 8.0.19 package), so SQLite tests behave identically. No schema change, no engine-boundary change, no wire-contract change.

## Implementation approach

1. REPOSITORY LAYER (Infrastructure). Follow PromoCodeRepository.TryIncrementRedemptionCountAsync exactly: `db.Database.ExecuteSqlInterpolatedAsync($\"\"\"...\"\"\", ct)` returning affected rows, fresh `Guid.NewGuid()` written to RowVersion in every statement. Physical names verified in 20260518135004_InitialCreate.cs: UsagePeriods, UsageReservations, RewriteAttempts, RewriteCredits. Statements (no date comparisons in SQL — dates are only WRITTEN as parameters, so no IsSqlite() branch is needed; use CASE WHEN, not GREATEST, for SQL Server 2022 + SQLite portability):
   a. UsagePeriodRepository.TryReserveSlotAsync(periodId, quotaLimit, now): UPDATE UsagePeriods SET ReservedCount = ReservedCount + 1, QuotaLimit = {quotaLimit}, UpdatedAt = {now}, RowVersion = {g} WHERE Id = {periodId} AND UsedCount + ReservedCount < {quotaLimit}. NOTE: guard uses the COMMAND quotaLimit, not the stored column — this preserves the current comparison `usedCount + reservedCount >= command.QuotaLimit` (CreateRewriteAttemptHandler.cs:80, ReserveQuotaHandler.cs:64) and the SET preserves PrepareUsagePeriodAsync's QuotaLimit refresh.
   b. UsagePeriodRepository.RefreshQuotaLimitAsync(periodId, quotaLimit, now): unconditional SET QuotaLimit/UpdatedAt/RowVersion WHERE Id — parity with PrepareUsagePeriodAsync on the credit path.
   c. UsagePeriodRepository.FinalizeReservedSlotAsync(periodId, now): SET ReservedCount = CASE WHEN ReservedCount > 0 THEN ReservedCount - 1 ELSE 0 END, UsedCount = UsedCount + 1, UpdatedAt = {now}, RowVersion = {g} WHERE Id = {periodId} (parity with Math.Max(0,...) at ProcessRewriteJobHandler.cs:267-268).
   d. UsagePeriodRepository.ReleaseReservedSlotAsync(periodId, now): same as (c) without the UsedCount increment.
   e. RewriteCreditRepository.TryConsumeForReservationAsync(creditId): UPDATE RewriteCredits SET AmountConsumed = AmountConsumed + 1, RowVersion = {g} WHERE Id = {creditId} AND AmountConsumed < AmountGranted. (Expiry is filtered at candidate selection, exactly as today — GetUsableForReservationAsync has no recheck either; keeping dates out of SQL avoids SQLite TEXT-comparison pitfalls.)
   f. RewriteCreditRepository.ReleaseConsumedAsync(creditId): SET AmountConsumed = CASE WHEN AmountConsumed > 0 THEN AmountConsumed - 1 ELSE 0 END, RowVersion = {g} WHERE Id = {creditId}.
   g. RewriteCreditRepository.ListUsableForReservationIdsAsync(userId, now): load user credits AsNoTracking, apply the EXACT in-memory filter+ordering currently in GetUsableForReservationAsync lines 33-38 ((ExpiresAt null or > now) && AmountGranted - AmountConsumed > 0; expiring credits first, then ExpiresAt asc, then GrantedAt asc), return ordered List<Guid>. Extract one private static helper used by both methods.
   h. UsageReservationRepository.TryTransitionFromPendingAsync(reservationId, target, now): UPDATE UsageReservations SET Status = {targetName}, FinalizedAt|ReleasedAt = {now}, RowVersion = {g} WHERE Id = {reservationId} AND Status = 'Pending'. Status is stored as string via HasConversion<string> (AppDbContext.cs:111) — pass nameof(UsageReservationStatus.X) as interpolated parameters. Throw ArgumentOutOfRangeException unless target is Finalized, Released, or Expired.
2. RESERVE FLOW — CreateRewriteAttemptHandler.HandleCoreAsync and ReserveQuotaHandler.ReserveCoreAsync (keep both in lockstep; public signatures, DTOs, and result kinds unchanged). Keep: user lookup/suspension (Create only), idempotency replay/conflict check, ComputeRequestHash, CreatePendingAttempt, outbox message creation (MessageType \"RewriteJobCreated\", CorrelationId=attemptId — pinned by QuotaUseCaseTests:17-57). Replace the quota block with:
   a. period = usagePeriods.GetByUserIdAndPeriodKeyAsync(...) (existing tracked read; the entity is NEVER mutated through EF anymore — tracked-but-unmodified is not written by SaveChanges).
   b. bool wonPeriodSlot = period is null ? command.QuotaLimit > 0 : (await usagePeriods.TryReserveSlotAsync(period.Id, command.QuotaLimit, command.Now, ct)) == 1. (A not-yet-existing period trivially has 0+0 capacity; the row is INSERTED later with ReservedCount=1, protected by unique index IX_UsagePeriods_UserId_PeriodKey — insert races surface as UK violations which UnitOfWork.IsRetryableDbUpdateRaceException already retries, UnitOfWork.cs:95-97.)
   c. Guid? consumedCreditId = null. If !wonPeriodSlot: foreach id in credits.ListUsableForReservationIdsAsync(userId, now) → if TryConsumeForReservationAsync(id) == 1 { consumedCreditId = id; break; }. If still null → return QuotaExceeded() — CRITICAL: at this point NOTHING has been added to the ChangeTracker and the failed conditional UPDATEs matched 0 rows, preserving the pinned no-pending-mutation invariant (QuotaUseCaseTests:152-188 asserts period.UpdatedAt/QuotaLimit untouched and zero rows after a post-handler SaveChanges).
   d. Only after a slot or credit is secured: if period is null, create the UsagePeriod entity (QuotaLimit=command.QuotaLimit, ReservedCount = wonPeriodSlot ? 1 : 0, CreatedAt/UpdatedAt=now) and AddAsync it; if period exists and the CREDIT path won, await usagePeriods.RefreshQuotaLimitAsync(period.Id, command.QuotaLimit, command.Now, ct) (parity with old PrepareUsagePeriodAsync).
   e. AddAsync attempt; AddAsync UsageReservation { UsagePeriod = newPeriod (navigation) OR UsagePeriodId = period.Id, RewriteCreditId = consumedCreditId, Status = Pending, CreatedAt = now, ExpiresAt = now + TTL }; AddAsync outbox row; SaveChangesAsync; return Created.
   f. Transaction wrapper: unitOfWork.ExecuteInTransactionAsync(core, IsolationLevel.ReadCommitted, ReservationRaceMaxAttempts /*3*/, ct). Justification to put in a code comment: the admission check now executes inside the UPDATE under row locks (loser re-evaluates the predicate after the winner commits — same mechanism as PromoCodes redemption, which already runs against Azure SQL with RCSI); Serializable only protected the old SELECT-then-write window and added deadlock surface; remaining first-insert and idempotency races are resolved by unique indexes + the existing retry classifier; Microsoft.Data.Sqlite promotes ReadCommitted to Serializable so SQLite-backed tests are unaffected (verified against Microsoft.Data.Sqlite 8.0.19: BeginTransaction(ReadCommitted) returns effective Serializable, no throw).
3. FINALIZE FLOW — ProcessRewriteJobHandler.FinalizeSuccessAsync and FinalizeQuotaSuccessHandler (identical bodies): inside ExecuteInTransactionAsync(ReadCommitted, maxAttempts:3): read attempt (tracked) + reservation (tracked, never EF-mutated); keep the two early-return guards exactly as today (Succeeded+Finalized → false; attempt Failed/Expired → false; ProcessRewriteJobHandler.cs:252-262); then claimed = TryTransitionFromPendingAsync(reservation.Id, Finalized, now) == 1; if !claimed return false; if reservation.RewriteCreditId is null → FinalizeReservedSlotAsync(reservation.UsagePeriodId, now); set attempt Succeeded/ResultJson/CompletedAt/RowVersion via EF; SaveChanges; return true.
4. RELEASE FLOW — ProcessRewriteJobHandler.ReleaseAsync and ReleaseQuotaHandler (identical bodies): inside ExecuteInTransactionAsync(ReadCommitted, maxAttempts:3): read reservation first (ids only); claimed = TryTransitionFromPendingAsync(reservation.Id, Released, now) == 1; if claimed → ReleaseConsumedAsync(creditId) or ReleaseReservedSlotAsync(periodId, now). THEN read the attempt (ORDERING RULE — the claim UPDATE blocks on a concurrent finalizer's row lock, so a post-claim read observes the finalizer's committed attempt status; reading before the claim could overwrite Succeeded with Failed now that Serializable no longer serializes whole transactions); if attempt.Status is Pending/Processing → set Failed + ErrorCode + CompletedAt + new RowVersion; SaveChanges. Residual races are caught by the RowVersion concurrency token (DbUpdateConcurrencyException is retryable per UnitOfWork.cs:86, and ChangeTracker.Clear() on retry flushes any stale identity-map instance so the rerun re-reads fresh state and no-ops).
5. EXPIRED CLEANUP — ReleaseExpiredReservationsHandler: keep the batch loop and ListExpiredPendingBatchAsync unchanged; switch the per-batch transaction to ReadCommitted (single attempt, timer/redelivery retries as today). Per reservation: claimed = TryTransitionFromPendingAsync(res.Id, Expired, command.Now) == 1; if !claimed continue; counter refund via ReleaseConsumedAsync / ReleaseReservedSlotAsync raw UPDATEs (do NOT mutate the tracked RewriteCredit/UsagePeriod/UsageReservation entities — only the RewriteAttempt entity is EF-mutated: Status=Expired, ErrorCode = Processing ? \"processing_timed_out\" : \"reservation_expired\", CompletedAt, new RowVersion); accumulate releasedCount from successful claims; keep loop termination on listedBatch.Count < BatchSize. The existing drain test (5 reservations, batch 2, mixed error codes) must pass unchanged.
6. DO NOT TOUCH: MarkQuotaProcessingHandler / MarkProcessingAsync (no counter involvement, already a guarded single-row transition), IRewriteEngineClient / providers / ServiceCollectionExtensions DI (engine-swap boundary, constraint 5), any Functions/frontend/wire contract (402 QuotaExceeded ProblemDetails, error codes, statuses all unchanged), IUnitOfWork (all needed overloads exist, IUnitOfWork.cs / UnitOfWork.cs:58-83).
7. TESTS: (a) new QuotaConcurrencyTests using a file-backed SQLite fixture with WAL + busy_timeout=30000 cloned from PromoConcurrencyTests.cs:410-470, one fresh AppDbContext + real repositories + real UnitOfWork per parallel task (CreateReserveHandler pattern from QuotaUseCaseTests:362-371 but per-context); (b) repository tests in InfrastructureRepositoryTests; (c) run the FULL suite — AdminRouteMetadataTests and InfrastructureServiceCollectionTests are unaffected but only the full run proves it.

## Risks

- SQL Server semantics under RCSI (Azure SQL default): conditional UPDATEs take update locks and re-evaluate predicates against the latest committed version, which is exactly what the one-winner guarantee relies on — this is the same mechanism PromoCodeRepository.TryIncrementRedemptionCountAsync already uses in production, but the new tests prove it only on SQLite (single-writer); the CI sqlserver-migration gate runs migrations, not these queries. Mitigation: identical statement shape to the proven promo path.
- Raw SQL hardcodes table/column names (UsagePeriods, RewriteCredits, UsageReservations, status strings 'Pending'/'Finalized'/'Released'/'Expired'); a future entity rename or HasConversion change would break at runtime, not compile time. Names verified against 20260518135004_InitialCreate.cs and AppDbContext.cs:104-131; the new repository tests pin them.
- Identity-map staleness: a tracked RewriteAttempt cached from MarkProcessingAsync in the same DI scope is NOT refreshed by a later tracked query; a stale guard read could attempt a wrong transition. Healed deterministically by the RowVersion concurrency token (write fails) + UnitOfWork retry path calling ChangeTracker.Clear() (UnitOfWork.cs:52,77) so the rerun reads fresh and no-ops — this is why finalize/release MUST use the maxAttempts:3 overload and why the release flow MUST read the attempt after the reservation claim.
- Behavioral edge intentionally preserved, not fixed: the expired-cleanup error-code reason (processing_timed_out vs reservation_expired) is computed from the batch-read attempt status and could in rare races be one transition stale; it is informational only and matches today's exposure.
- Sequencing change on the credit path: candidates are consumed first-success-wins in order, so under concurrency a loser now falls to the NEXT usable credit instead of retrying the whole Serializable transaction — strictly fewer 500s, but credit pick order under simultaneous contention can interleave (total consumption is still exact).
- Worker/ServiceBusRewriteWorker.cs calls the same ProcessRewriteJobHandler (prior audit flagged it not line-audited); it inherits the change automatically and is not deployed, but is compiled in CI — no separate edit expected.
- FreeBaselineZero users (QuotaLimit=0): the new-period + zero-limit branch must still INSERT the UsagePeriod row when a credit wins (UsageReservation.UsagePeriodId is a non-nullable FK, AppDbContext.cs:117-120); covered by RewriteApiTests.V1_rewrite_submit_allows_live_key_with_usable_purchased_credit.
- Unknown until full-suite run: other tests may construct these handlers with bespoke IUnitOfWork fakes that record isolation levels; repo grep found none pinning Serializable, but the SimulatedRetryUnitOfWork pattern (QuotaUseCaseTests:412-457) shows fakes implement the full interface — any new fake compile errors are confined to the test project.

## Design notes

Adjacent problems noticed but deliberately OUT of this issue's scope: (1) CreateRewriteAttemptHandler and ReserveQuotaHandler are near-duplicates — consolidation is a separate cleanup; (2) MarkProcessingAsync/MarkQuotaProcessingHandler keep Serializable read-modify-write on the attempt row (no counters involved; safe via RowVersion) — could become a conditional UPDATE later; (3) DeleteAccountHandler and RetentionService touch attempts/credits outside the reservation lifecycle and were not changed; (4) ReleaseExpiredReservationsHandler still loads full entities with three Includes when only IDs + attempt status are needed — a projection would cut I/O; (5) GetUsableForReservationAsync loads ALL user credits into memory (kept as-is for the ordering-pin test; ListUsableForReservationIdsAsync reuses the same helper rather than fixing the scan). Key empirical verification done during design: Microsoft.Data.Sqlite 8.0.19 BeginTransaction(IsolationLevel.ReadCommitted) succeeds and reports effective Serializable — so the isolation downgrade cannot break any SQLite-backed test. Isolation choice: ReadCommitted (not RepeatableRead/Snapshot) because every correctness-bearing decision now happens inside single UPDATE statements guarded by row locks plus unique indexes (IX_UsagePeriods_UserId_PeriodKey, IX_RewriteAttempts_UserId_IdempotencyKey) whose violations the existing UnitOfWork classifier retries; nothing in the flow depends on read stability across statements after the ordering rule (claim before attempt read) is applied. Codex brief should include CLAUDE.md's five banned substrings as a constraint and explicitly forbid touching ServiceCollectionExtensions.cs, Providers/**, and any Functions HTTP surface.
