# HARD-03: Stripe webhook ingest-then-process

> Priority: P0 · Adversarial verdict: **REVISE** · Migration: **YES** — additive EF migration required
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Split Stripe webhook handling into ingest-then-process. The HTTP path (StripeWebhookFunction) keeps signature verification byte-identical, then only upserts a StripeEvent row in a new Pending status with the raw payload persisted, and returns 200. Processing moves to a background path — a new ProcessPendingStripeEventsHandler driven by a 15s timer function (mirroring the existing outbox dispatcher claim/lease pattern on the existing Status/LockedUntil/AttemptCount columns) with exponential backoff and a terminal poisoned Failed state plus a structured alert log. A bounded inline fast-path (default 8s budget, env-disable-able) still attempts processing inside the request via the same claim primitive, preserving today's checkout-grant latency and most existing test behavior; any inline failure leaves the event Pending for the timer instead of returning 5xx to Stripe.

## Implementation approach

1. STATUS MODEL (existing columns only, per mandate). Pending = ingested/awaiting work; LockedUntil doubles as not-before time (null = due now). Processing = claimed; LockedUntil = lease expiry (keep today's 2-minute lease, StripeEventRepository.cs:40). Processed = terminal success; PayloadJson set to null (scrub). Failed = TERMINAL poison only (after MaxAttempts) or missing-payload. Existing index (Status, LockedUntil) at AppDbContext.cs:139 already serves the claim query.

2. ENTITY + MIGRATION. Add nullable string PayloadJson to Domain/Entities/StripeEvent.cs. Add Pending=3 to StripeEventStatus (string-converted column — purely additive). Generate EF migration AddStripeEventPayloadJson (Infrastructure project/context AppDbContext, startup per repo convention). Nothing dropped/renamed.

3. REPOSITORY (StripeEventRepository.cs + IStripeEventRepository.cs). Remove BeginProcessingAsync (its callers are deleted). Add:
   - Task<IReadOnlyList<StripeEvent>> ClaimDueAsync(DateTimeOffset now, int batchSize, TimeSpan lease, string? eventId, CancellationToken ct): claim rows where (Status==Pending AND (LockedUntil == null OR LockedUntil <= now)) OR (Status==Processing AND LockedUntil <= now), optional EventId filter for the inline single-event path, ORDER BY CreatedAt ASC, Take(batchSize); for each: Status=Processing, LockedUntil=now+lease, AttemptCount+=1, LastAttemptAt=now, RowVersion=Guid.NewGuid(). MUST have an IsSqlite() branch doing the date filters in memory, copied from OutboxMessageRepository.ClaimDueAsync (lines 24-47).
   - void MarkRetryScheduled(StripeEvent e, string error, DateTimeOffset nextAttemptAt, DateTimeOffset now): Status=Pending, LockedUntil=nextAttemptAt, LastError=truncate(error,1000), ProcessedAt=null, RowVersion=new.
   - MarkProcessed additionally sets PayloadJson=null. MarkFailed stays as-is (terminal poison: Status=Failed, LockedUntil=null) and KEEPS PayloadJson for diagnosis.

4. INGEST HANDLER (IngestStripeWebhookHandler). One Serializable transaction via unitOfWork.ExecuteInTransactionAsync: load by EventId (GetByEventIdAsync). (a) null → AddAsync new StripeEvent { EventId, Type, Status=Pending, AttemptCount=0, PayloadJson=RawBody, CreatedAt=Now, LockedUntil=null } → Accepted. (b) Status==Processed → AlreadyProcessed, no writes. (c) Status==Pending or Failed → refresh Type and PayloadJson=RawBody, Status=Pending, LockedUntil=null (a fresh Stripe delivery makes the event immediately due; re-arms poisoned events on manual dashboard resend, granting exactly one fresh attempt because the poison check fires again at AttemptCount>=MaxAttempts), keep AttemptCount → AlreadyPending. (d) Status==Processing with LockedUntil>Now → leave untouched (active lease) → AlreadyPending. Catch stripeEvents.IsDuplicateEventWriteFailure(ex) (concurrent insert race, StripeEventRepository.cs:85-92) → AlreadyPending.

5. PROCESSOR (ProcessPendingStripeEventsHandler, constructor: IStripeEventRepository, StripeEventPayloadSynchronizer, IRewriteCreditRepository, IUnitOfWork, StripeEventProcessingOptions, ILogger<>). Mirrors DispatchDueOutboxHandler exactly: claim phase = ExecuteInTransactionAsync(Serializable, maxAttempts:5) wrapping ClaimDueAsync + SaveChanges. Then per claimed event sequentially (CreatedAt order from the claim):
   a. If PayloadJson is null → MarkFailed(e, "payload_missing", now) in a short tx, LogError poison (see 5e), continue. (Covers pre-migration straggler rows in Processing with expired leases.)
   b. Parse payload: new StripeWebhookPayloadDto(e.EventId, e.Type, e.PayloadJson) (existing ctor, StripeWebhookPayloadDto.cs:11). Parse failure (JsonException) → treat as sync failure (step 5d).
   c. Success path: open Serializable tx → run synchronizer.SyncAsync(payload, now, postCommitActions, ct) (logic identical to old ProcessStripeWebhookHandler.SyncPayloadAsync — moved verbatim, including checkout grant idempotence via credits.ExistsByStripeEventIdAsync + UK on RewriteCredit.StripeEventId, invoice upsert, grace state, refunds/disputes). If SyncAsync returns null → stripeEvents.MarkProcessed(e, now) → SaveChanges → commit → run postCommitActions outside the tx, each wrapped in try/catch; on exception LOG WARNING \"{PaymentObservabilityEvent} stripe_event_notification_failed\" with EventId/Type (replaces today's silent swallow at ProcessStripeWebhookHandler.cs:533-547). Count this event as processed.
   d. Failure path (SyncAsync returned a failure string, or any non-cancellation exception): in a fresh short tx reload the event; if AttemptCount >= options.MaxAttempts → stripeEvents.MarkFailed (poison) + step 5e; else MarkRetryScheduled with nextAttemptAt = now.AddMinutes(Math.Min(60, Math.Pow(2, e.AttemptCount))) (2,4,8,16,32,60,60... min; ~3h horizon at MaxAttempts=8).
   e. Poison alert: logger.LogError(\"{PaymentObservabilityEvent} Stripe event {EventId} of type {EventType} poisoned after {AttemptCount} attempts: {Error}\", \"stripe_event_poisoned\", ...) — structured and queryable in App Insights traces; poisoned rows also surface through the existing readiness count HealthFunction.CountFailedStripeEventsAsync (HealthFunction.cs:223-226), which now becomes a pure poison signal.
   f. Checkout-grant duplicate race: catch credits.IsStripeEventIdWriteFailure(ex) for type checkout.session.completed and mark the event Processed (move MarkProcessedAfterCheckoutGrantConflictAsync from ProcessStripeWebhookHandler.cs:455-489 verbatim).

6. HTTP FUNCTION (StripeWebhookFunction.cs). Lines 27-116 (body read, secret check → 500, signature check → 400, id/type extraction → 400) stay byte-identical. Replace lines 118-139 with: (i) ingest via IngestStripeWebhookHandler — exceptions here keep today's behavior: LogError \"webhook_failed\" + rethrow → 500 so Stripe redelivers (we have no durable record yet); (ii) inline fast-path: if result != AlreadyProcessed AND options.InlineBudgetSeconds > 0, create linked CTS with CancelAfter(InlineBudgetSeconds), call processor.HandleAsync(new(UtcNow, BatchSize:1, EventId:eventId)); catch ALL exceptions including the budget cancellation (but not request-ct cancellation), LogWarning \"{PaymentObservabilityEvent} webhook_inline_deferred\" — the event stays Pending (or lease-held Processing that the timer reclaims after <=2min); (iii) return OkObjectResult(new { received = true, processed }) where processed = inline pass returned >0; AlreadyProcessed → processed=false (preserves the pinned duplicate-replay body \"processed\":false, StripeWebhookApiTests.cs:52). FAST-PATH DECISION: inline is kept because it is the exact same DB-bound work the endpoint already does synchronously today (no new external calls; the only outbound call, the billing-portal URL inside StripeEventNotifier.ResolveBillingPortalUrlAsync, already self-falls-back, StripeEventNotifier.cs:86-106), it preserves checkout-credit-grant latency for the user returning from Stripe Checkout, and it is now failure-isolated: the 200 no longer depends on processing success. The 8s budget keeps the response inside Stripe's delivery timeout; STRIPE_WEBHOOK_INLINE_BUDGET_SEC=0 turns the endpoint into pure ingest.

7. ORDERING DECISION. Claim order is CreatedAt ASC; the timer trigger is a host-singleton and the batch is processed sequentially → near-FIFO. No per-customer ordering barrier is added: (a) every per-event handler writes absolute state taken from the event payload (subscription status sync, invoice upsert keyed by invoice id, credit grant idempotent by UK on RewriteCredit.StripeEventId), (b) the same interleaving already exists in production because Stripe redelivers failed events hours-to-days later while newer events for the same customer flow through, and (c) the daily Stripe reconciliation run is the backstop. A backoff-delayed older event can therefore be overtaken by a newer one — accepted, pre-existing exposure, not a regression. Strict per-customer serialization would require a CustomerId column on StripeEvent (listed in notes, out of scope).

8. TIMER FUNCTION. StripeEventProcessorTimerFunction: TimerTrigger \"*/15 * * * * *\", calls processor with (UtcNow, BatchSize:10, EventId:null); log count when >0. Timer functions are not HTTP routes → AdminRouteMetadataTests route pin unaffected.

## Risks

- Behavioral change: after a successful ingest the endpoint returns 200 even when processing fails, so Stripe's own redelivery no longer drives retries — our timer owns them. If the Functions host were down, Pending events would accumulate silently; mitigated because the same host serves the HTTP trigger (ingest would fail too → 500 → Stripe retries) and poison events hit the existing Failed-count readiness check, but there is no alert on aged-Pending backlog (noted as adjacent work).
- Retry horizon shrinks from Stripe's ~3 days of redelivery to ~3 hours (8 attempts, capped 60-min backoff). The orphan-checkout race (event before user row) almost always resolves in seconds, but a >3h-late resolution now requires manual dashboard resend (which re-arms the event) or reconciliation. Knob STRIPE_EVENT_MAX_ATTEMPTS exists if this proves too tight.
- Same-customer ordering remains best-effort: an event in backoff can be overtaken by a newer event. Verified pre-existing (Stripe redelivery causes identical interleaving today; handlers write absolute state; credits idempotent by UK) — accepted, not a regression, documented in the design.
- Monitoring semantics shift: StripeEventStatus.Failed counts (HealthFunction.cs:223-226) drop because transient failures now park as Pending; Failed becomes poison-only. Intended, but any external dashboard keyed on Failed counts will read lower.
- PayloadJson stores the raw Stripe payload (customer ids, amounts) at rest until processed; scrubbed on Processed but retained on Failed for diagnosis — no purge job for old poisoned payloads (noted as adjacent retention work).
- Inline budget cancellation mid-transaction leaves the row lease-held (Processing) for up to the 2-minute lease before the timer reclaims it — worst-case extra latency for that event, no correctness impact.
- Api/Program.cs test-host route must be changed in lockstep with the Functions route or WebApplicationFactory-based suites diverge from production behavior.
- StripeEventUseCaseTests churn (~15 tests share one helper): the mechanical helper rewrite is low-risk but a Codex worker must NOT alter the moved business-logic methods while relocating them — the diff for StripeEventPayloadSynchronizer.cs should be move-only.
- TryMarkStripeEventProcessedHandler appears registered but unused outside its own test (verified by grep) — deliberately left untouched; do not let the worker 'clean it up' in this PR.
- Unknown: whether APPLICATIONINSIGHTS_CONNECTION_STRING is set on the live Function app (not verifiable from source) — the stripe_event_poisoned LogError alert depends on it for queryability.

## Design notes

Verified-by-reading facts this design rests on: StripeEvent has NO payload column today (Domain/Entities/StripeEvent.cs:5-17) — that is why the migration is unavoidable; Status is persisted as a string (AppDbContext.cs:136) so adding the Pending enum member is data-safe; the (Status, LockedUntil) index already exists (AppDbContext.cs:139); the claim/lease/backoff pattern to mirror is OutboxMessageRepository.ClaimDueAsync + DispatchDueOutboxHandler (5-retry Serializable claim, 30s lease, per-message try/catch); the existing webhook handler runs everything synchronously in the HTTP request and rethrows on failure → non-200 (StripeWebhookFunction.cs:118-137), with post-commit notification exceptions swallowed silently (ProcessStripeWebhookHandler.cs:533-547). Engine-swap constraint #5: this design touches nothing under Infrastructure/Providers, Domain/RewriteEngine, or the IRewriteEngineClient boundary. Adjacent problems intentionally left OUT of scope: (1) no aged-Pending backlog readiness check (HealthFunction only counts Failed) — a one-line CountReadinessCheck addition later; (2) no retention purge for PayloadJson on poisoned rows; (3) TryMarkStripeEventProcessedHandler is dead code outside its test; (4) strict per-customer FIFO would need a CustomerId column on StripeEvent + per-customer claim grouping; (5) subscription coverage gap in daily reconciliation (HARD-scope elsewhere) is unchanged; (6) the timer functions still have no end-to-end trigger test (house-wide gap — handlers are tested instead, consistent with OutboxDispatcherTimerFunction). Env knobs introduced (names only, optional, parse-with-default at registration): STRIPE_EVENT_MAX_ATTEMPTS (default 8), STRIPE_WEBHOOK_INLINE_BUDGET_SEC (default 8, 0 disables the inline fast-path).
