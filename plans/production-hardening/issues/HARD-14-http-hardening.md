# HARD-14: HTTP hardening: size caps, error envelope, correlation

> Priority: P2 · Adversarial verdict: **PASS** · Migration: No
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Add a single Functions-worker middleware (HttpHardeningMiddleware) that enforces a uniform ~64KB request-body cap (413 + frozen coded envelope, larger configurable cap for the Stripe webhook), resolves/validates/echoes X-Correlation-Id on every HTTP endpoint and pushes it into the logging scope; unify the error envelope by keeping the two contract-frozen shapes ({error:{code,message}} for coded errors — the dominant machine-readable shape on the public v1 surface — and ProblemDetails for consumer/admin) while additively injecting a machine-readable "code" field into all ProblemDetails responses through the single FunctionHttpResults choke point; add baseline security headers (CSP report-only, X-Content-Type-Options, Referrer-Policy, Permissions-Policy) via next.config.ts headers(). No engine, quota, or schema changes; no EF migration.

The frontend never parses ProblemDetails bodies (app/api/rewrite/route.ts:224-229 passes non-OK bodies through verbatim and branches only on HTTP status + attempt errorCode; no .title reads anywhere in app/ or components/app except unrelated DOM usage). STEP 1 — HttpHardeningMiddleware.cs (new, namespace ReplyInMyVoice.Functions.Http). Class: public sealed class HttpHardeningMiddleware(IConfiguration configuration, ILogger<HttpHardeningMiddleware> logger) : IFunctionsWorkerMiddleware. Constants: public const string CorrelationHeaderName = "X-Correlation-Id"; public const long DefaultMaxRequestBodyBytes = 65_536; public const long DefaultStripeWebhookMaxRequestBodyBytes = 1_048_576; env names "HTTP_MAX_REQUEST_BODY_BYTES" and "HTTP_MAX_REQUEST_BODY_BYTES_STRIPE_WEBHOOK" (read by NAME via configuration[...], long.TryParse with default, clamp to minimum 1024 — never print values; follows the raw-config house pattern).
Invoke(FunctionContext context, FunctionExecutionDelegate next):
  var http = context.GetHttpContext(); if (http is null) { await next(context); return; }  // timers/ServiceBus untouched
  var correlationId = ResolveCorrelationId(http.Request);
  http.Items["CorrelationId"] = correlationId;
  http.Response.Headers[CorrelationHeaderName] = correlationId;   // set pre-execution; IActionResult execution later does not clear headers
  var limit = ResolveBodyLimitBytes(configuration, context.FunctionDefinition.Name);
  if (await TryRejectOversizedBodyAsync(http, limit, context.CancellationToken)) { return; }  // short-circuit: do NOT call next
  using var scope = logger.BeginScope(new Dictionary<string, object> { ["CorrelationId"] = correlationId });
  await next(context);
Public static testable core (public, NOT internal — Functions has no InternalsVisibleTo; only Infrastructure does):
  public static string ResolveCorrelationId(HttpRequest request): take request.Headers[CorrelationHeaderName].ToString(); valid iff length 1..64 and every char matches [A-Za-z0-9._-]; otherwise Guid.NewGuid().ToString("D").
  public static long ResolveBodyLimitBytes(IConfiguration config, string functionName): functionName == "StripeWebhook" (exact name from StripeWebhookFunction.cs:21 [Function("StripeWebhook")]) → HTTP_MAX_REQUEST_BODY_BYTES_STRIPE_WEBHOOK default 1MB; else HTTP_MAX_REQUEST_BODY_BYTES default 64KB.
  public static async Task<bool> TryRejectOversizedBodyAsync(HttpContext http, long limit, CancellationToken ct): (a) if Request.ContentLength is { } len && len > limit → write 413 and return true; (b) if request can carry a body (ContentLength > 0, or ContentLength null && method is POST/PUT/PATCH/DELETE) → buffer-read Request.Body into a MemoryStream copying AT MOST limit+1 bytes (loop with a fixed 8KB buffer; stop as soon as copied > limit); if copied > limit → write 413, return true; else memory.Position = 0; Request.Body = memory; return false. This deterministic pre-buffering (max 64KB, 1MB only for Stripe which already ReadToEndAsync's the body at StripeWebhookFunction.cs:27-28) deliberately avoids exception-based stream guards, because BillingHttpFunctions.cs:111 and similar broad catches would otherwise convert a mid-read overflow into a 500.
  public static string BuildPayloadTooLargeJson(long limit): JsonSerializer.Serialize of new { error = new { code = "payload_too_large", message = $"Request body must be {limit} bytes or smaller." } } — byte-compatible with the frozen coded envelope.
  Private WritePayloadTooLargeAsync: http.Response.StatusCode = StatusCodes.Status413PayloadTooLarge; ContentType = "application/json"; WriteAsync(BuildPayloadTooLargeJson(limit)). The correlation header is already on the response.

## Risks

- First IFunctionsWorkerMiddleware in this repo: the short-circuit pattern (write to HttpContext.Response, skip next) is the documented ASP.NET-integration approach, but no host-level Functions test harness exists to exercise it end-to-end; unit tests cover the static core only. Mitigate post-merge: curl the deployed /api/health for the X-Correlation-Id echo and POST an oversized body to /api/rewrite expecting 413 (manual smoke, CI /api/health+/api/version gates already in dotnet-azure.yml).
- CorrelationId log-scope flow into Application Insights depends on ApplicationInsightsLoggerOptions.IncludeScopes (default true) AND on APPLICATIONINSIGHTS_CONNECTION_STRING being set on the live Function app — the latter was not verifiable from source (carried risk from the guardrails audit).
- Middleware buffers request bodies up to limit+1 bytes in memory per request (max 64KB; 1MB only for StripeWebhook, which already buffers via ReadToEndAsync at StripeWebhookFunction.cs:27). Negligible at current scale but listed for transparency.
- A >1MB Stripe event (very rare, e.g. enormous invoice line arrays) would get 413; Stripe retries with backoff and the daily reconciliation report (StripeReconciliationTimerFunction) surfaces missed grants; the cap is adjustable via the HTTP_MAX_REQUEST_BODY_BYTES_STRIPE_WEBHOOK app setting without a deploy.
- 413 now fires before API-key auth on v1 (oversized request with an invalid key gets 413 not 401) — uniform transport backstop semantics; SDK/MCP treat any non-202 submit as failure so no client contract breaks.
- next.config.ts headers() on OpenNext/Cloudflare: redirects() in the same file already works in prod (proof the routing-manifest path is honored), but per project memory CF-layer changes have only been truly verifiable in a deployed environment — CSP is Report-Only precisely so a gap breaks nothing; verify response headers on prod after deploy.
- Adding Extensions['code'] to ProblemDetails responses is additive JSON; verified no frontend code parses ProblemDetails bodies and StripeBillingApiTests deserializes tolerantly — residual risk only for unknown external consumers of the consumer API (none known; site is pre-launch traffic).
- Unknown/not line-audited: whether any AdminHttpFunctions endpoint wraps its body read in a broad catch — irrelevant to the chosen design (buffering happens in middleware before the function runs), listed because it killed the alternative throwing-stream design.

## Design notes

Envelope decision rationale: the two shapes are BOTH contract-frozen ({error:{code,message}} by openapi.json+SDK+MCP on v1; ProblemDetails by the consumer contract, e.g. 402 'Rewrite quota exhausted'), so full convergence to one shape is impossible without breaking a frozen surface. Unification here = one choke point (FunctionHttpResults) + every error body machine-readable (coded envelope as-is; ProblemDetails gains additive 'code') + all NEW transport errors use the coded envelope. Adjacent problems found but deliberately OUT of scope: (1) consumer Next proxies (app/api/rewrite etc.) build fresh NextResponses and do not forward X-Correlation-Id nor send one upstream — a follow-up could generate the id at the Worker edge and pass it through; (2) StripeWebhookFunction/BillingHttpFunctions keep their private ResolveCorrelationId with TraceIdentifier fallback (log-only; could later read HttpContext.Items['CorrelationId']); (3) empty-body NotFoundResult/NoContentResult responses are not envelopes and were left alone (many layer-2 tests assert `is NotFoundResult`); (4) HealthFunction's {ok:...} shape is pinned by the CI health gate and Azure probes — exempt; (5) rewrite-workspace.tsx titleForQualityFailure checks stale reason strings ('signal_unavailable'/'fact_check_failed') that never match real codes — pre-existing, untouched; (6) no CSP report-uri collector exists yet (Sentry DSN pending per project status) — wire reporting once SENTRY_DSN lands. Engine-swap safety (constraint 5): everything sits at the Functions HTTP transport layer and next.config — no contact with IRewriteEngineClient, FactReconstructRewriteProvider, providers, DI provider selection, or ResultJson. Quota invariant: a 413 short-circuits before CreateRewriteAttemptHandler, so no reservation is ever created, let alone charged. Codex brief reminder: never print secret values; env vars by NAME only; banned substrings must not appear in any added line (the delivery-driver Gate1 grep is diff-scoped over app/components/public/lib).
