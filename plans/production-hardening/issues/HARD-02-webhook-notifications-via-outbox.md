# HARD-02: Stripe post-commit notifications through outbox

> Priority: P0 · Adversarial verdict: **REVISE** · Migration: No
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Replace the fire-and-forget post-commit notification delegates (exceptions silently swallowed) in ProcessStripeWebhookHandler, ProcessPaymentGraceRemindersHandler, and ProcessExpiredPaymentGraceHandler with OutboxMessage rows written inside the same UnitOfWork transaction as the state change, using four new message types (PaymentFailedNotification, PaymentRecoveredNotification, SubscriptionPausedNotification, PaymentGraceReminderNotification). The existing 15s OutboxDispatcherTimerFunction delivers them via new IOutboxMessageHandler implementations that load the user and invoke the unchanged StripeEventNotifier, inheriting the outbox retry/backoff/MaxAttempts=10 policy. Terminal outbox failure becomes visible via a new IOutboxDispatchObserver: Failed row status (already exists) + ILogger.LogError + App Insights TrackEvent. No schema change — the OutboxMessage table already exists.

## Risks

- Delivery semantics change from at-most-once (inline, lossy) to at-least-once: a dispatcher crash between StripeEventNotifier send and MarkSentAsync re-sends the email on the next 15s tick. Accepted for notifications; a NotificationDedupe key was deliberately not added to stay additive-zero on schema.
- Notification latency grows from ~0 to up to ~15s (outbox tick) plus backoff on transient failures; Resend/Stripe-portal hiccups now retry up to 10x over ~25 min before terminal Failed. This is the intended durability tradeoff.
- Dispatch-time user reload can diverge from enqueue-time state: PaymentGraceReminder self-suppresses via the existing PaymentGraceEndsAt null-check in StripeEventNotifier.cs:52-56, but PaymentRecovered/SubscriptionPaused will send based on the committed decision even if the user state changed in the intervening seconds — same exposure window the old post-commit code had, now slightly longer.
- TelemetryClient is only registered in the Functions host (Functions/Program.cs:22); in Api/test hosts the observer is logger-only. Whether APPLICATIONINSIGHTS_CONNECTION_STRING is set on the live Function app was not verifiable from source (pre-existing unknown from the guardrails audit), so the LogError + DB Status=Failed row are the guaranteed signals; the custom event is additionally subject to host.json adaptive sampling (low volume, practically retained).
- ProcessStripeWebhookHandler atomicity relies on a structural convention: notification AddAsync calls must stay on return-null paths of Sync* methods so the MarkFailed branch (HandleAsync:46-51) never flushes a staged notification row. The new sync-failure test pins this, but future edits to Sync* methods could reintroduce the hazard.
- ServiceBusRewriteWorker/OutboxDispatcherWorker (Worker project, built-but-not-deployed) resolves DispatchDueOutboxHandler from DI — compiles unchanged, but was not line-audited at runtime depth (pre-existing risk noted in the audit).
- StripeEventNotifier.EnqueueFailedPaymentNotificationAsync / PaymentGraceReminder create a live Stripe billing-portal session at dispatch time (timer context instead of webhook request context); behavior and its existing /app-URL fallback are unchanged, but Stripe API calls now originate from the dispatcher — relevant if a per-call Stripe rate concern ever applies to the 15s loop.
- Adding Microsoft.ApplicationInsights 2.23.0 to Infrastructure widens its package surface; version is pinned to match the Functions WorkerService transitive (2.23.0) to avoid NU1605/diamond conflicts in the single test project that references both.

## Design notes

Scope interpretation: the mandate names ProcessStripeWebhookHandler but lists grace-reminder among the message types, and grace-reminder is only produced by ProcessPaymentGraceRemindersHandler (the identical swallowed-post-commit pattern, lines 83-97); ProcessExpiredPaymentGraceHandler produces subscription-paused the same way. All three are included to satisfy 'no notification may be lost silently anymore'; anything narrower would leave two of the four mandated message types lossy. Naming: PascalCase MessageTypes (PaymentFailedNotification etc.) follow the existing house convention ('RewriteJobCreated', Ordinal dictionary in DispatchDueOutboxHandler.cs:14-15) rather than the mandate's kebab-case examples, which were prefixed 'e.g.'. Engine-swap safety (constraint 5): nothing here touches IRewriteEngineClient, providers, ResultJson, error codes, or provider HttpClients; the observer hooks the generic outbox dispatcher, which also gives RewriteJobCreated terminal failures visibility for free. ADJACENT PROBLEMS, deliberately out of scope: (1) ProcessExpiredPaymentGraceHandler's post-commit CancelSubscriptionAsync is still fire-and-forget with swallowed exceptions — a lost cancellation keeps billing a downgraded user; it needs its own outbox message type plus Stripe-side idempotency and deserves a separate issue; (2) CreditExpiryNotifier, TaxTurnoverNotifier, and StripeReconciliationNotificationAlerter send inline with similar loss modes; the same outbox pattern applies; (3) terminal-Failed outbox rows have no admin requeue surface (visibility only via log/metric/DB); (4) no test exercises OutboxDispatcherTimerFunction end-to-end (pre-existing gap, handler-level coverage only); (5) the StripeWebhookFunction HTTP path still runs the whole Serializable transaction synchronously inside the request — out of HARD-02's mandate (that is the inbox/processing concern, not notification delivery). Secrets: no env values read or printed; the observer logs message metadata only, never PayloadJson (payload contains only userId/occurredAt anyway — no PII beyond an internal GUID).
