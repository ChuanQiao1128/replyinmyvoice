## Summary

HARD-13 lands three data-hygiene fixes in the C# backend. (a) ApiKey.CurrentPeriodUsage is verifiably DEAD code today — no reader or writer exists anywhere in src/ or tests/ (only the entity declaration ApiKey.cs:16 and migration snapshots); per-key usage is already computed from ApiKeyUsage rows (ApiKeyUsageRepository.CountByApiKeyAsync, surfaced as Last30dUsage in ApiKeyHttpFunctions:105) and rate limiting from ApiKeyRateLimitWindow (ApiKeyRateLimiter) — so the fix is to mark the property [Obsolete] with a pointer to the real sources, keep the column mapped (additive-only), and pin both facts with reflection tests. (b) Add an EF Core global query filter RewriteAttempt.DeletedAt == null in AppDbContext, with IgnoreQueryFilters() at the 11 audited call sites that legitimately need deleted rows (job finalize/release, idempotency replay, expired-reservation release, webhook claim/enqueue, cost logging, retention scrub, account/admin erasure, sandbox idempotency lookups); the two Include(x => x.RewriteAttempt) sites (UsageReservationRepository.ListExpiredPendingBatchAsync, WebhookDeliveryRepository.ClaimDueAsync) are mandatory or soft-deleting an in-flight attempt would leak quota / fail webhooks. (c) Extend RetentionService with PurgeExpiredSandboxAttemptsAsync that hard-deletes RewriteAttempt rows belonging to IsTest API keys (joined via ApiKeyId, defense-in-depth "test:" idempotency-key prefix) older than 7 days, wired into the existing RetentionPurgeFunction timer. No external contract changes; no EF migration.

## Design notes

Verification highlights that corrected the prior audit's framing: (a) is NOT a dual-write problem today — grep proves ApiKey.CurrentPeriodUsage has no writer and no reader in any C# code (only the entity declaration and migration designer artifacts; the old Prisma schema under lib/generated/prisma still names it but that is Slice-7 dead TS). The /api/v1/usage endpoint computes from AccountUsagePlan (UsagePeriod/credits), console per-key usage from ApiKeyUsage rows, and rate limiting from ApiKeyRateLimitWindow — so the mandate's \"compute on read\" option is already the live behavior and the fix is purely to seal the dead column. HasQueryFilter is not recorded in EF migration snapshots and [Obsolete] is not a schema change, hence migration=false; the dotnet-azure.yml sqlserver-migration gate passes trivially. Adjacent problems found but OUT OF SCOPE (do not expand): (1) ApiKey.CurrentPeriodStartedAt is equally dead and could get the same [Obsolete] treatment; (2) ApiKeyRateLimitWindow rows accumulate forever (one per key per active minute) with no purge — a natural future RetentionService extension; (3) AppUser soft/hard delete has no equivalent filter (erasure model instead); (4) the sandbox purge retention (7d) is a constant, not an env knob — if ops wants tuning, follow the options-from-env pattern in AddReplyInMyVoiceInfrastructure, not IOptions; (5) RetentionService still loads full scrub/purge row sets without batching — same as existing behavior, acceptable at current volume; (6) Api/Program.cs:1236 carries a TODO(DDD) to move the inline sandbox db code into a handler (DDD-67/68) — the IgnoreQueryFilters edit keeps the duplicate in sync but the dedup itself belongs to the DDD wave. Engine-swap safety (constraint 5): every touched file is outside FactReconstructRewriteProvider and its strategies/gates; the only engine-adjacent edit is one IgnoreQueryFilters in RewriteCostLogger's attempt lookup, which sits on the persistence side of the boundary and survives any engine replacement.
