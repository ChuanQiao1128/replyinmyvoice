# HARD-01: Engine swap boundary: freeze contract + seal leaks

> Priority: P0 (foundational — land first) · Adversarial verdict: **REVISE** · Migration: No
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Freeze the rewrite-engine boundary that already exists (IRewriteEngineClient + Domain.Contracts.RewriteRequest) and make it the only coupling surface: add a canonical error-code constants class in Domain, seal the two source-level leaks (provider seam types declared inside FactReconstructRewriteProvider.cs; AsyncLocal metric capture declared inside OpenAiCompatibleRewriteModelClient.cs) by verbatim same-namespace file moves, add backend xUnit + frontend vitest contract tests that pin the minimal ResultJson shape and error-code set consumed by ProcessRewriteJobHandler, v1 API, webhooks, cost logger and the Next proxies, and write docs/rewrite-engine-contract.md with the swap recipe. No behavior change; the current engine passes untouched; no EF migration.

## ⚠️ Required corrections (adversarial review — apply these, they OVERRIDE the approach text where they conflict)

"rewrite_model_failed"`, passing through model-client codes model_http_<status>/model_timeout/model_empty/model_candidate_missing/model_json_parse_failed/model_network_failed (backend-dotnet/src/ReplyInMyVoice.Infrastructure/Providers/OpenAiCompatibleRewriteModelClient.cs:37-73) and model_not_configured (UnavailableRewriteModelClient, wired at ServiceCollectionExtensions.cs:272). Fix in the design text (no code change): define failure ErrorCode as an OPEN set that consumers must tolerate (they already do), make EngineEmittable the canonical/recommended set for new engine implementations, and explicitly document the current model_* / rewrite_model_failed pass-through family in both IRewriteEngineClient's XML-doc and docs/rewrite-engine-contract.md section 4.
- Test spec for EngineContract_error_code_constants_pin_wire_values is self-contradictory: it says 'asserts every const equals its exact snake_case wire string and EngineEmittable membership', but 6 of the 13 constants (ProviderJsonParseFailed, RequestJsonParseFailed, ReservationExpired, ProcessingTimedOut, PolicyIntentGateFailed, EngineUnavailableFallback) are intentionally NOT in EngineEmittable. A zero-context Codex worker implementing it literally gets a red test whose cheapest 'fix' is adding all constants to the frozen set, corrupting the contract. Respecify as: assert each const's exact wire string, assert QualityGateNotCharged == the exact 5-code set, and assert EngineEmittable == the exact 7-code set (exhaustive equality on both frozen sets, not blanket membership).

- ProcessRewriteJobHandler literal→constant swap is behavior-identical only if constant values are exact; the wire values are pinned by EngineContract_error_code_constants_pin_wire_values plus existing RewriteJobUseCaseTests/RewriteApiTests
- The vitest source-string extraction regex may be brittle to future reformatting of the Set literal in the proxy routes — acceptable: a formatting change that breaks extraction forces a human look at the frozen set, which is the point of a pin
- Sapling/writing-signal calls are never recorded into ProviderCalls today (cost log hardcodes Sapling*=0) — this design documents but does not fix it (fix requires engine-internal changes, deferred to the swap); cost rows for a future engine depend on it following the documented ProviderCalls rule
- DeterministicRewriteProvider (keyless dev/test composition) emits success JSON without naturalness, so v1 reports its succeeded attempts as engine_unavailable — pre-existing behavior, now pinned and documented rather than changed; do not 'fix' it in this PR (it would alter InfrastructureServiceCollectionTests-pinned dev behavior)
- Worker/ReplyInMyVoice.Worker/ServiceBusRewriteWorker.cs:87 resolves the same ProcessRewriteJobHandler — unaffected by constants adoption, but the Worker project is built in CI, so the solution must compile end to end (covered by acceptance #1)
- policy_intent_gate_failed stays in the frozen not-charged set despite never being emitted by the current engine — intentional additive contract; removing it would be a breaking contract narrowing
