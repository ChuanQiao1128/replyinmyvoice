# HARD-07: Fix provider circuit-breaker state lifetime + fast-fail

> Priority: P1 · Adversarial verdict: **PASS** · Migration: No
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Hoist the provider circuit-breaker state out of ProviderHttpResilienceHandler (currently a private nested class whose state dies when HttpClientFactory rotates handler chains every ~2min) into a per-provider-name singleton registry, so circuit state survives rotation and is shared across concurrent chains. When open, the handler throws ProviderCircuitOpenException (derived from HttpRequestException) before touching the network; both provider clients already catch HttpRequestException, so failures surface instantly through the EXISTING error codes (model_network_failed via FactReconstructRewriteProvider.cs:167, quality_signal_unavailable via :106) and the existing never-charge release path (ProcessRewriteJobHandler.cs:153-167) — no engine, Application, Functions, or frontend changes. Thresholds become env-configurable with current values as defaults, state changes emit logs plus a metric hook interface for HARD-11, and timeouts (caller-token cancellations) are now recorded as circuit failures so a hanging provider can actually trip the breaker. Decision: keep the hand-rolled handler; do NOT migrate to Microsoft.Extensions.Http.Resilience (justification in notes).

## Implementation approach

2. CREATE Resilience/IProviderResilienceEvents.cs:
   `public interface IProviderResilienceEvents { void CircuitStateChanged(string providerName, ProviderCircuitState fromState, ProviderCircuitState toState, int sampledFailures, int sampledTotal); }`
   plus `public sealed class NoOpProviderResilienceEvents : IProviderResilienceEvents` (empty body) and `public static class ProviderResilienceEventNames { public const string CircuitStateChanged = "provider_circuit_state_change"; }` — the constant is the metric/event name HARD-11's TelemetryClient implementation must use (dimensions: provider, from_state, to_state, failure_count, sample_count).

3. CREATE Resilience/ProviderCircuitBreaker.cs:
   - `public enum ProviderCircuitState { Closed, Open, HalfOpen }`
   - `public sealed class ProviderCircuitOpenException : HttpRequestException` with `string ProviderName` property; message "Provider HTTP circuit is open for {providerName}.". MUST derive from HttpRequestException so OpenAiCompatibleRewriteModelClient.cs:71-74 (-> model_network_failed) and SaplingWritingSignalClient.cs:78 (-> unavailable signal -> quality_signal_unavailable at FactReconstructRewriteProvider.cs:106/209) keep their existing mappings — this is what guarantees fast-fail lands on EXISTING error codes with quota released and never charged.
   - `public readonly record struct CircuitLease(bool IsProbe);`
   - `public sealed class ProviderCircuitBreaker` ctor `(string providerName, ProviderCircuitBreakerOptions options, ILogger logger, IProviderResilienceEvents events, Func<DateTimeOffset>? clock = null)` (clock defaults to () => DateTimeOffset.UtcNow; injectable for tests). All state under one private lock: `Queue<(DateTimeOffset ObservedAt, bool Success)> _samples`, `ProviderCircuitState _state`, `DateTimeOffset? _openUntil`, `DateTimeOffset? _probeStartedAt`.
   - `public CircuitLease Acquire()`: Closed -> lease(IsProbe=false). Open & now < _openUntil -> throw ProviderCircuitOpenException. Open & now >= _openUntil -> transition to HalfOpen (log Info + event), set _probeStartedAt=now, return lease(IsProbe=true). HalfOpen: if _probeStartedAt is set and now - _probeStartedAt <= BreakDuration -> throw (another probe outstanding); else grant a fresh probe lease (lease-expiry safety net against a probe that never records).
   - `public void ThrowIfOpen()`: throws only when (state==Open && now < _openUntil) or (state==HalfOpen with an unexpired outstanding probe). Used by retry attempts > 0 so a circuit opened mid-retry by other callers fails fast.
   - `public void Record(CircuitLease lease, bool success)`: if lease.IsProbe: success -> Closed, clear _samples and _probeStartedAt (log Info + event HalfOpen->Closed); failure -> Open with _openUntil = now + BreakDuration, clear _probeStartedAt (log Warning + event HalfOpen->Open). If not probe: only applies when state==Closed — evict samples older than SamplingDuration (same logic as current RecordCircuitResult at :533-556), enqueue, and when count >= MinimumThroughput and failures/count >= FailureRatio set state=Open, _openUntil = now + BreakDuration (log Warning "Provider circuit opened for {Provider}: {Failures}/{Total} failures in sampling window; breaking for {BreakSeconds}s." + event Closed->Open).
   - `public ProviderCircuitState State { get; }` (lock-read, for tests/metrics). No secret values anywhere; provider name is the HttpClient logical name only.

## Risks

- Behavior delta 1 (intentional): caller-token cancellations (per-call timeouts) now count as circuit failures. A burst of legit cancellations (e.g., host shutdown mid-flight) could contribute samples; mitigated by min-throughput 8 within 30s, and without this change a hanging provider can never open the circuit (verified: IsTransientException at ServiceCollectionExtensions.cs:587-590 excludes canceled tokens, and the exception currently propagates unrecorded), making fast-fail unreachable in the exact 180s-budget-burn scenario the mandate targets.
- Behavior delta 2 (intentional): current code fully closes the circuit and clears samples when the break expires (:528-529); the new design is a stricter single-probe half-open, so during provider recovery concurrent requests fail fast until one probe succeeds. Exposure bounded by PROVIDER_CIRCUIT_BREAK_SEC.
- Sapling outages will now actually trip a working circuit, so consumers see 422 quality_signal_unavailable (not charged) within ms instead of slow 10s-timeout failures — correct per product rules but a visible UX shift during Sapling incidents.
- Circuit state is per-process (singleton), not distributed: with multiple Function app instances each opens independently — same as the intended design today, just actually working; documented, not fixed here.
- The <1s fast-fail-through-the-engine claim depends on current engine internals having no inter-attempt sleeps (verified: FactReconstructRewriteProvider.cs:302-335 has no Task.Delay). A swapped engine keeps the HTTP-level guarantee (every call on a named resilient client fails instantly when open) regardless, because all new code sits at the HttpClient/DI layer and ProviderCircuitOpenException is an HttpRequestException any HTTP client already handles.
- Unknown (not runtime-verified): whether HttpClientFactory's 2-minute handler rotation was the only reset path in the live Functions host (scale-out instance recycling also resets state); the fix addresses both within one process, but production validation needs App Insights logs after deploy (the new state-change log lines make this observable for the first time).
- RewriteProviderAdapterTests' existing 429 test goes through the new wiring; if the worker deviates from the specified Acquire/Record placement (e.g., records intermediate retry failures), the default thresholds could be hit by test traffic — the spec deliberately records only terminal outcomes, matching current :484-502 behavior.
- InfrastructureServiceCollectionTests pins DI registrations; this design only ADDS registrations (options, events, registry) and changes no existing service types, but the worker must not rename or remove the two AddResilientProviderHttpClient client names — nameof(OpenAiCompatibleRewriteModelClient)/nameof(SaplingWritingSignalClient) are the registry keys.

## Design notes

DECISION — keep the hand-rolled handler, do not migrate to Microsoft.Extensions.Http.Resilience: (1) the only defect is breaker-state lifetime; the retry layer (Retry-After aware, buffered request replay, jitter) is correct and behavior-pinned by RewriteProviderAdapterTests; (2) M.E.Http.Resilience would solve state sharing (its pipelines are keyed per client name) but replaces BOTH retry and circuit semantics, adds a new NuGet dependency to Infrastructure, and still requires mapping BrokenCircuitException — a strictly larger diff for the same outcome; (3) the registry hoist is ~250 new lines, mostly moved code. Re-evaluate Polly only if a future item needs hedging/rate-limiter pipelines. HARD-11 coordination: implement IProviderResilienceEvents with the worker-registered TelemetryClient (Functions/Program.cs:22-23) using ProviderResilienceEventNames.CircuitStateChanged = "provider_circuit_state_change" (dimensions: provider, from_state, to_state, failure_count, sample_count); registration point is the TryAddSingleton in ServiceCollectionExtensions. Adjacent problems noticed but OUT of scope: (a) Sapling calls are never recorded into RewriteProviderCallCapture so cost rows hardcode Sapling=0 (OpenAiCompatibleRewriteModelClient.cs:321ff is the only Record site); (b) named provider HttpClients keep the 100s default HttpClient.Timeout, redundant behind the per-call linked CTS timeouts; (c) IRewriteModelClient/IWritingSignalClient are still defined inside FactReconstructRewriteProvider.cs and referenced by DI (boundary leak flagged by the engine audit; moving them belongs to the engine-swap prep item, not here); (d) ValidateReplyInMyVoiceRuntimeConfiguration does not validate the new circuit env vars — deliberate, they are optional tuning knobs with safe defaults. Operating-mode note: per CLAUDE.md supervisor discipline, implementation should be delegated to a Codex worker as one PR; this design is self-contained for that brief.
