# HARD-05: Reconciliation: close the loop

> Priority: P0 · Adversarial verdict: **PASS** · Migration: **YES** — additive EF migration required
> Part of the production-hardening wave. Read `../SPEC.md` first for the engine-swap-boundary principle and the frozen external contract.

## Summary

Upgrade the daily Stripe reconciliation from an in-memory dry-run report to a closed loop: (a) auto-grant missing PURCHASE credits for paid-but-ungranted PaymentIntents (capped per run, idempotent via the existing filtered UK on RewriteCredit.StripeEventId using a deterministic synthetic event id, with an AdminAuditLog row whose DetailsJson carries source=reconciliation); (b) persist a StripeReconciliationRun row every run and emit the discrepancy alert through the house outbox pattern (new MessageType handled by a new IOutboxMessageHandler that delegates to the existing email alerter); (c) add a subscription pass comparing Stripe subscription status vs AppUser.SubscriptionStatus in both directions, report-only (never auto-revoke, and grant-no-payment also stays report-only). Window lookback becomes configurable (default 3 days) so payments skipped by the min-age guard are retried on later runs.

6. HANDLER REWRITE (ReconcileStripeHandler.cs). New ctor: `(IPaymentGrantRepository, IStripePaymentReconciliationClient, IRewriteCreditRepository credits, IAppUserRepository appUsers, IAdminUserRepository adminUsers, IStripeReconciliationRunRepository runs, IOutboxMessageRepository outboxMessages, StripeReconciliationOptions options, IUnitOfWork unitOfWork)` — remove IStripeReconciliationAlerter. Flow:
   a. Validate window (unchanged). Build the payment report exactly as today (BuildReport untouched).
   b. AUTO-GRANT PASS — for each PaidButNoGrant discrepancy ordered by StripePaidAt asc: if `options.AutoGrantMaxPerRun == 0` skip all with reason `over_cap`… no: when cap is 0, classify all as `over_cap` manual-review. If payment.PaidAt > command.CompletedAt.AddMinutes(-options.MinPaymentAgeMinutes) → manual-review `payment_too_recent` (will be retried next run thanks to the 3-day window). Else call `FindCheckoutSessionForPaymentIntentAsync`; classify: null → `no_checkout_session`; Mode != "payment" or PaymentStatus != "paid" → `not_payment_mode`; rewrites = session.GrantedRewrites > 0 ? that : sku map {quick_pack:10, value_pack:30} (exact mirror of ProcessStripeWebhookHandler.ResolveGrantedRewrites:622-635) else `rewrites_unresolvable`; user = `appUsers.FindForStripeCheckoutAsync(session.ExternalAuthUserId, session.CustomerId)`; null → `user_not_found`. If eligible and grantedSoFar < AutoGrantMaxPerRun: execute ONE Serializable `unitOfWork.ExecuteInTransactionAsync` per grant containing: (i) idempotency re-check `credits.ExistsByStripeEventIdAsync($"reconciliation:{paymentIntentId}")` OR `(await credits.ListByStripePaymentIntentIdAsync(paymentIntentId)).Any(c => c.Source == "PURCHASE")` → skip silently (count nothing); (ii) `credits.AddAsync(new RewriteCredit { UserId = user.Id, Source = "PURCHASE", AmountGranted = rewrites, OriginalAmountGranted = rewrites, AmountConsumed = 0, GrantedAt = command.CompletedAt, ExpiresAt = command.CompletedAt.AddDays(90), StripeEventId = $"reconciliation:{paymentIntentId}", StripePaymentIntentId = paymentIntentId, StripeSku = session.Sku, StripeAmountTotal = payment.AmountReceived (fallback session.AmountTotal), StripeCurrency = payment.Currency })` — field-for-field parity with the webhook grant at ProcessStripeWebhookHandler.cs:141-156 except StripeEventId/ReceiptUrl; (iii) `adminUsers.AddAuditLogAsync(new AdminAuditLog { AdminExternalAuthUserId = "system:reconciliation", AdminEmail = string.Empty, Action = "reconciliation_auto_grant", TargetUserId = user.Id, DetailsJson = <camelCase JSON containing source:"reconciliation", paymentIntentId, creditId, sku, rewrites, windowStart, windowEnd>, CreatedAt = command.CompletedAt })`; (iv) SaveChanges. Catch non-cancellation exceptions per grant → manual-review `grant_write_failed`, continue. Eligible beyond cap → manual-review `over_cap`, counted in AutoGrantSkippedCount. GrantButNoPayment and AmountMismatch rows are NEVER mutated — no revoke path exists in this handler (assert in test).
   c. SUBSCRIPTION PASS — `stripeSubs = ListSubscriptionsAsync()`; `localUsers = ListSubscriptionUsersForReconciliationAsync()`. Map Stripe status with the same table as ProcessStripeWebhookHandler.MapSubscriptionStatus:597-611 (private copy with a comment pointing at the source). Active-ish bucket = {Active, Trialing, PastDue}. Direction 1: each Stripe sub whose mapped status is active-ish — locate local user by StripeSubscriptionId then StripeCustomerId; if missing or local status not active-ish (treat Inactive and Canceled as the same non-active bucket) → `stripe_active_local_not`. Direction 2: each local active-ish user — if StripeSubscriptionId is null, or not present in stripeSubs, or its Stripe status maps non-active → `local_active_stripe_not`. Both directions report-only.
   d. PERSIST + ALERT in one final Serializable transaction: `runs.AddAsync(new StripeReconciliationRun { WindowStart, WindowEnd, StartedAt = command.CompletedAt (or a StartedAt captured by the timer), CompletedAt, StripePaymentCount, PurchaseGrantCount, PaidButNoGrantCount, GrantButNoPaymentCount, AmountMismatchCount, SubscriptionMismatchCount, AutoGrantedCount, AutoGrantSkippedCount, ManualReviewCount, ReportJson = camelCase-serialized report with each list truncated to first 200 entries })`; if `DiscrepancyCount > 0 || AutoGrantedCount > 0 || SubscriptionMismatchCount > 0 || ManualReviewCount > 0` also `outboxMessages.AddAsync(new OutboxMessage { MessageType = "StripeReconciliationAlertRequested", PayloadJson = same report JSON, Status = Pending, CreatedAt/NextAttemptAt = command.CompletedAt, MaxAttempts = 10, CorrelationId = run.Id.ToString() })` (mirror CreateRewriteAttemptHandler.CreateRewriteJobOutboxMessage:176-189); SaveChanges. Return the report.

## Risks

- Stripe key posture: ApplicationStripeBillingClient.CreateStripeClient (ApplicationStripeBillingClient.cs:147-157) throws 'stripe_test_key_required' unless STRIPE_SECRET_KEY starts with sk_test_, and the reconciliation list call already flows through it — if the live Function app holds a live key, today's reconciliation timer fails nightly and the new closed loop will too. Unverifiable from source; the design mirrors the existing guard rather than changing key policy (changing it is a separate owner decision).
- Stripe.net 48.5.0 / pinned API 2025-08-27.basil: SessionListOptions.PaymentIntent and SubscriptionListOptions.Status='all' are assumed available (standard in this SDK line) — verified only at build time, not from source here; Subscription.CurrentPeriodEnd is deliberately NOT read because Basil moved it off the subscription object.
- Subscription invoice PaymentIntents appear in ListPaidPaymentIntentsAsync and have no payment-mode checkout session, so each Pro/API renewal will surface as PaidButNoGrant + manual-review no_checkout_session — pre-existing false-positive noise that this design contains (never auto-grants) but does not eliminate; suppressing it needs an invoice-linkage signal the current DTO lacks.
- Cross-path race: reconciliation auto-grant and a concurrent webhook delivery use different StripeEventId values, so the UK cannot catch a simultaneous double insert; both paths now check-by-payment-intent inside Serializable transactions, leaving only a tiny race window on SQL Server (range locks on the indexed lookups serialize in practice). A filtered unique index on StripePaymentIntentId was rejected because existing prod data cannot be verified duplicate-free from source.
- ListSubscriptionsAsync(Status='all') is unwindowed; with many historical subscriptions the daily call grows — acceptable at current scale, revisit with a created-window filter if subscription volume grows.
- Widening the window to 3 days re-reports unresolved discrepancies on consecutive runs (by design, idempotent) — alert emails may repeat until a human clears the underlying issue.
- AdminAuditLog requires AdminExternalAuthUserId/AdminEmail; the synthetic identity 'system:reconciliation' + empty email will appear in any admin audit UI — cosmetic, precedent exists (GrantCreditsHandler already writes string.Empty email).
- Report/payload size: ReportJson and the outbox PayloadJson are nvarchar(max) but lists are truncated to 200 entries to bound row size; a catastrophic discrepancy day loses list detail beyond 200 (counters remain exact).
