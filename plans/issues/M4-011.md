GitHub: https://github.com/ChuanQiao1128/replyinmyvoice/issues/196
Milestone: M4-Landing

## Task

Redesign the customer-visible frontend so Reply In My Voice no longer feels like a rough template. Use the installed `$web-design-engineer` skill as the design lead.

## Required Skills

- Read and follow `/Users/qc/.codex/skills/web-design-engineer/SKILL.md`.
- If the current Codex session has not indexed the skill, read the file directly by path and follow it as the fallback.
- Because this is browser-visible frontend work, follow the repo `ui-browser-testing` route from `AGENTS.md` where applicable.
- Append required entries to `docs/skill-run-log.md` before finishing.

## Scope

Redesign the high-impact customer-facing surfaces first:

- `/` landing page and landing components.
- `/pricing`.
- `/sign-in` and `/sign-up` pages if they are still visible in the current auth flow.
- Shared header/footer/navigation.
- `/app` rewrite workspace visual shell, without changing rewrite/auth/billing behavior.

Do not redesign admin-only pages unless a shared component change naturally affects them.

## Design Direction

Start by inspecting the existing UI, CSS tokens, landing components, app workspace, typography, colors, spacing, and responsive behavior.

Choose a direction appropriate for this product:

- quiet, polished, trustworthy communication product;
- for teachers, salespeople, workplace email, and customer replies;
- not detector-bypass and not generic AI SaaS;
- avoid purple/blue gradient AI aesthetics, decorative orbs, emoji icons, nested cards, and stock-looking hero visuals.

```markdown
- Typography:
- Layout:
- Shape and surfaces:
- Motion:
- Assets:
```

Also include a current-state critique score using the `$web-design-engineer` five dimensions.

## Implementation Requirements

- Prefer existing Next.js, Tailwind, component, and route patterns.
- Use structured tokens/classes instead of scattered one-off colors.
- Keep card radius at 8px or less unless an existing component already requires otherwise.
- Do not nest cards inside cards.
- Use `lucide-react` icons where icons are appropriate.
- Preserve legal/product copy accuracy: no fabricated testimonials, metrics, customer logos, claims, or screenshots.
- Preserve all existing auth, rewrite, quota, billing, API, telemetry, and webhook behavior.
- Keep public copy aligned with product positioning: "Replies that still sound like you."
- Add or update focused tests only where useful for stable UI copy/contracts.

## Required Critique Gate

After implementation, run critique mode from `$web-design-engineer` and record results in `plans/frontend-redesign-design-brief.md`:

- Philosophy alignment: 0-10
- Visual hierarchy: 0-10
- Craft quality: 0-10
- Functionality: 0-10
- Originality: 0-10

Target: average score >= 8.0 and no dimension below 7.0.

If the score misses target, iterate at least once on the highest-impact issues before deciding whether to commit. If the remaining work is too large for one safe PR, create `plans/frontend-redesign-followups.md` with scoped follow-up issues and leave this issue with a materially improved first pass.

## Browser Verification

Before `ready_to_commit`, verify:

- `npm run lint`
- `npm run typecheck`
- focused or full tests appropriate to changed files
- local browser/Playwright check for desktop and mobile viewport on `/`, `/pricing`, and an `/app` workspace preview or the real route if auth allows
- no console errors, major layout overlap, clipped text, off-screen controls, or broken assets

If browser verification is blocked by local environment limits, document exactly what failed and keep the code-level validation green.

## Acceptance

- The redesigned frontend is visibly more polished, consistent, and product-specific.
- First viewport clearly communicates Reply In My Voice and leaves a hint of the next section visible.
- `/app` remains a usable working surface, not a marketing page.
- Mobile and desktop layouts are both checked.
- Final design critique score is recorded with evidence.
