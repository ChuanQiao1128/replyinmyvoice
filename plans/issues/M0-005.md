# M0-005: Verify GitHub Actions CI is green on main

**Milestone**: M0-Stabilize
**Est**: ≤15 min
**Depends on**: M0-004 (local green)

## TASK
After M0-002 PR merges and M0-004 lands, confirm GitHub Actions workflows pass on `main`. If a workflow fails, diagnose and either fix or ticket.

## CONTEXT
- Workflows: `.github/workflows/cloudflare-worker.yml`, `.github/workflows/dotnet-azure.yml`
- These run on push to main; M0-002's merge will trigger them
- `dotnet-azure.yml` may have its own infra needs — if it fails on Azure credential issues, that's a known M1+ scope concern, ticket and skip

## CONSTRAINTS
- Read-only diagnosis: read action logs via `gh run list` + `gh run view <id> --log`
- Fix only trivial issues (env var name typo, etc.); larger → ticket
- Don't modify Azure deploy logic in M0 — that belongs to M1 / a later milestone
- Don't trigger manual workflow_dispatch runs unnecessarily

## CHANGES REQUIRED
1. `gh run list --branch main --limit 5` — find the latest runs
2. For each failing run: `gh run view <id> --log-failed` to see error
3. Categorize: trivial inline-fix vs ticket-for-later
4. Inline fixes: branch + PR as usual
5. For deferred: `gh issue create --milestone <appropriate> --title "..."`
6. Write `plans/m0-ci-status.md` recording: latest run IDs, statuses, fixes/tickets

## ACCEPTANCE
- `cloudflare-worker.yml` passes on main (build, test, deploy steps)
- `dotnet-azure.yml` either passes or has a follow-up issue with the failure logged
- Status report written

## DO NOT
- Push to main directly
- Touch deploy credentials
- Disable a failing workflow (ticket instead)
- Use force-push or any destructive git operation

## On completion
Comment with run URLs and final status. Close when CI is green or all failures are ticketed.
